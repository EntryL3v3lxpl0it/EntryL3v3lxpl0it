# External Network Penetration Testing Methodology

**Document type:** Repeatable assessment playbook
**Scope of applicability:** Internet-facing assets explicitly listed in an authorized Statement of Work / Rules of Engagement (RoE)
**Standards referenced:** PTES, MITRE ATT&CK (Enterprise), CWE, CVSS v3.1/v4.0, NIST SP 800-115, OWASP WSTG (for exposed web surfaces)

> **Operating constraints baked into this methodology**
> - Every active action is bounded to in-scope, authorized targets. No testing of assets discovered but not authorized — log them as "out-of-scope observations" instead.
> - Default to non-destructive validation. Prove reachability/impact with the minimum action required; stop at the safe checkpoint defined per phase.
> - Authentication testing defaults to **password spraying with lockout awareness**, never blind brute force against lockout-enabled accounts.
> - Separate **evidence** (observed) from **hypothesis** (inferred). Findings ship only with evidence.
> - Rate-limit scanning to avoid availability impact; coordinate windows for anything that could degrade a production service.

---

## 0. Phase 0 — Pre-Engagement (Planning & Scoping)

**Goal:** Establish legal authorization, scope, RoE, and deconfliction before any packet is sent.

### 0.1 Authorization gate (hard blocker)
Do not proceed past this section until all are collected and stored in `00-scope/`:

1. Signed authorization / "get-out-of-jail" letter naming the testing entity, the authorizing party, and target ownership attestation.
2. In-scope target list: domains, FQDNs, IP addresses, CIDR ranges, ASN(s).
3. **Explicit exclusions** (IPs, hosts, paths, fragile devices, third-party/cloud-tenant assets requiring provider authorization).
4. RoE: testing windows, allowed intensity, prohibited techniques (e.g., DoS, social engineering, destructive payloads), data-handling rules.
5. Source IPs you will test from (for allow-listing / blue-team deconfliction).
6. Emergency contacts + stop-test procedure.
7. Confirmation of **who owns the IP space** (cloud-hosted ranges frequently require the cloud provider's testing policy to be honored).

### 0.2 Decision point
- If any target resolves to a third-party (CDN, SaaS, shared cloud) you do not have authorization for → **STOP** that target, escalate for written authorization.
- If RoE prohibits exploitation → cap the engagement at vulnerability validation (Phase 4) and document reachability only.

### 0.3 Scope normalization
Resolve and expand scope into a canonical target file. Capture both the authorized list and what it actually resolves to (record discrepancies as evidence).

###### Expand CIDRs and resolve FQDNs into a working list; keep raw + sorted/unique
###### (in-scope only — feed this file from 00-scope/authorized_targets.txt)
```bash
sort -u 00-scope/authorized_targets.txt > 00-scope/scope_canonical.txt
```

**Validation gate (exit Phase 0):** authorization stored, scope_canonical.txt exists, exclusions encoded, source IPs shared with client.

```bash

```
```bash
cd "$ENGAGEMENT"
```
```bash
echo "Engagement root: $(pwd)"
```

| Directory | Contents |
|-----------|----------|
| `00-scope/` | Authorization, RoE, canonical target lists, exclusions |
| `01-osint/` | Passive recon: subdomains, emails, leaked creds refs, cert data |
| `02-discovery/` | Host discovery + port scan outputs (masscan/nmap) |
| `03-enum/<svc>/` | Per-service enumeration, segregated by protocol |

## 2. Phase 1 — Passive Information Gathering (OSINT)

**Goal:** Build the attack surface picture **without touching client infrastructure** (zero-touch / light-touch). MITRE ATT&CK: Reconnaissance (TA0043).

> Zero-touch sends no traffic to the target. Light-touch should look like normal traffic. Keep authoritative lists of valid targets, people, and credential references as you go.

### 2.1 DNS & subdomain surface (passive sources first)

#### Passive subdomain aggregation (uses public datasets / passive sources)
```bash
subfinder -d client.tld -all -silent | tee 01-osint/subfinder.txt
```
```bash
amass enum -passive -d client.tld -o 01-osint/amass_passive.txt
```

###### Certificate transparency (passive)
######   crt.sh, censys, shodan — record findings into 01-osint/
###### DNS records + zone-transfer attempt (light-touch, authorized)
```bash
dnsrecon -d client.tld -t std -j 01-osint/dnsrecon.json
```
```bash
dnsenum client.tld | tee 01-osint/dnsenum.txt
```

Consolidate, then resolve only in-scope names:
```bash
cat 01-osint/subfinder.txt 01-osint/amass_passive.txt | sort -u > 01-osint/subdomains_all.txt
```
```bash
dnsx -l 01-osint/subdomains_all.txt -a -resp -silent | tee 01-osint/subdomains_resolved.txt
```

### 2.2 People, emails, exposure
```bash
theHarvester -d client.tld -b bing,duckduckgo,crtsh -f 01-osint/harvester
# Record (do not auto-query without authorization):
#   - Email format (hunter.io)
#   - Breach references (haveibeenpwned / dehashed / intelx) — REFERENCE only
#   - Employees (LinkedIn) for username modeling
```

```bash
spiderfoot -s client.tld -o csv > 01-osint/spiderfoot.csv   # if running SpiderFoot scans within RoE
```

### 2.3 Internet exposure mapping (third-party datasets)
Record from Shodan/Censys (passive): open ports, banners, TLS certs, technologies, exposed admin portals. Useful Shodan filters: `hostname:"client.tld"`, `net:"<CIDR>"`, `org:"<org>"`, `ssl.cert.subject.cn:"client.tld"`.

### 2.4 Outputs & validation gate
- `01-osint/subdomains_resolved.txt` — in-scope live names → IPs
- `01-osint/` exposure notes, username model, tech fingerprints
- **Evidence vs. hypothesis:** OSINT findings are *hypotheses* until confirmed by active testing. Mark them as such.
- **Gate:** confirmed in-scope host:IP:service candidate list ready for active discovery.

## 3. Phase 2 — Active Host Discovery & Port Scanning

**Goal:** Identify live hosts and open ports across the authorized ranges. MITRE ATT&CK: Active Scanning (T1595). NIST 800-115 "Discovery."

> External ranges frequently filter ICMP. Use a two-stage approach: fast breadth (masscan) → targeted depth (nmap). Throttle rate to protect availability.

### 3.1 Stage 1 — Breadth (fast port sweep)
```bash
# Wide TCP sweep, rate-limited. Tune --rate to RoE (start conservative).
masscan -iL 00-scope/scope_canonical.txt -p1-65535 --rate 1000 \
  --excludefile 00-scope/exclusions.txt -oG 02-discovery/masscan_tcp.gnmap

# Extract live host:port pairs
grep -E 'open' 02-discovery/masscan_tcp.gnmap \
  | awk '{print $4, $7}' | sort -u > 02-discovery/live_hostports.txt

# Live hosts and live ports (for nmap targeting)
grep /open/ 02-discovery/masscan_tcp.gnmap | cut -d' ' -f4 | sort -uV > 02-discovery/live_hosts.txt
grep /open/ 02-discovery/masscan_tcp.gnmap | grep -oE '[0-9]+/open' | cut -d/ -f1 | sort -un | paste -sd, > 02-discovery/live_ports.txt
```

### 3.2 Stage 2 — Depth (service/version + default scripts)
```bash
# Service + version detection and default NSE on the ports masscan found open.
# -Pn because external hosts often drop ICMP; do NOT add aggressive timing on fragile targets.
nmap -Pn -sV -sC \
  -p "$(cat 02-discovery/live_ports.txt)" \
  -iL 02-discovery/live_hosts.txt \
  --version-intensity 5 \
  -oA 02-discovery/nmap_services
```

UDP is slower and noisier — scope it to high-value ports only:
```bash
nmap -Pn -sU --top-ports 50 -iL 02-discovery/live_hosts.txt -oA 02-discovery/nmap_udp_top
```

### 3.3 Scan-type reference
| Flag | Scan | Use |
|------|------|-----|
| `-sS` | SYN (half-open) | Default fast TCP, needs raw socket |
| `-sT` | TCP connect | When unprivileged / through proxychains |
| `-sU` | UDP | DNS/SNMP/IKE/NTP/TFTP discovery |
| `-sV` | Version | Confirm service + version (drives vuln mapping) |
| `-sC` | Default NSE | Safe enumeration scripts |
| `-Pn` | Skip host discovery | External hosts dropping ICMP |
| `-oA` | All-format output | Always, for evidence parity |

### 3.4 Decision point
- Service identified with a version banner → route to the matching Phase 3 service module.
- Port open but unidentified → targeted `-sV --version-all` and banner grab (`nc`/`ncat`, `openssl s_client` for TLS) before classifying.
- Host newly appears or disappears between stages → note as evidence (possible IPS/rate-limiting); reduce rate and re-confirm.

**Validation gate:** `02-discovery/nmap_services.gnmap` enumerated; every open port mapped to a service+version or flagged "needs manual ID."

## 4. Phase 3 — Service Enumeration

**Goal:** For each exposed service, enumerate configuration, accessible data, auth surfaces, and version-specific weaknesses. Confirm behavior; distinguish confirmed from inferred. MITRE ATT&CK: T1595.002 / T1046.

Run the module that matches each open service. Each module lists: **entry point → enumeration → what to capture → safe stopping point.**

### 4.1 Web / HTTP(S) — ports 80/443/8080/8443/etc.
```bash
# Identify which hosts speak HTTP(S)
httpx -l 02-discovery/live_hosts.txt -p 80,443,8080,8443,8000 \
  -title -status-code -tech-detect -web-server -o 03-enum/web/httpx.txt

# Visual triage at scale
gowitness scan file -f 02-discovery/live_hosts.txt --screenshot-path 07-evidence/screenshots/web/
# or: eyewitness --web -f 03-enum/web/urls.txt -d 07-evidence/screenshots/web/

# WAF + fingerprint + TLS posture (per host)
wafw00f https://target/ | tee 03-enum/web/wafw00f_target.txt
whatweb -a3 https://target/ | tee 03-enum/web/whatweb_target.txt
testssl.sh --quiet --color 0 https://target/ > 03-enum/web/testssl_target.txt   # TLS/cipher/cert weaknesses

# Content & path discovery (rate-limited; respect RoE)
feroxbuster -u https://target/ -w /usr/share/wordlists/seclists/Discovery/Web-Content/raft-medium-directories.txt \
  -t 20 --rate-limit 50 -o 03-enum/web/ferox_target.txt
# or: ffuf -u https://target/FUZZ -w <wordlist> -mc 200,204,301,302,307,401,403 -o 03-enum/web/ffuf_target.json

# Known-misconfig / template-based checks
nikto -h https://target/ -o 03-enum/web/nikto_target.txt
```
**Capture:** server/tech stack, TLS findings (cert validity, weak ciphers, protocol support), exposed admin/login panels, default content, directory listings, interesting status codes.
**Safe stop:** enumerate and screenshot. Defer exploitation of any web app vuln to Phase 5 and, for deep app testing, hand off to the web-app methodology (OWASP WSTG).

### 4.2 SMB / NetBIOS — 139/445
```bash
nbtscan -r <CIDR> | tee 03-enum/smb/nbtscan.txt
enum4linux-ng -A target | tee 03-enum/smb/enum4linux_target.txt

# Null/guest session + share access (read-only listing)
smbmap -H target -u '' -p '' | tee 03-enum/smb/smbmap_null_target.txt
netexec smb target -u '' -p '' --shares | tee 03-enum/smb/nxc_null_target.txt

# Protocol/signing posture (relay exposure indicator)
netexec smb target | tee 03-enum/smb/nxc_signing_target.txt
```
**Capture:** OS/domain/workgroup, SMB version + signing status, accessible shares and contents, null-session exposure.
**Safe stop:** list shares and note readable/writable. Do not exfiltrate share contents beyond proof (filenames + counts, per the "don't be the breach" principle).

### 4.3 SMTP — 25/465/587
```bash
nmap -Pn -p25,465,587 --script smtp-commands,smtp-open-relay,smtp-ntlm-info target -oN 03-enum/smtp/nmap_smtp_target.txt
smtp-user-enum -M VRFY -U 01-osint/usernames.txt -t target | tee 03-enum/smtp/userenum_target.txt
# Open-relay confirmation (benign, to self-controlled mailbox only):
swaks --to test@yourdomain --from test@client.tld --server target   # observe accept/reject
```
**Capture:** banner, supported verbs (VRFY/EXPN), open-relay behavior, user-enumeration oracle, NTLM info leak.
**Safe stop:** confirm relay/enumeration once; do not send bulk or to third parties.

### 4.4 SNMP — 161/UDP
```bash
onesixtyone -c /usr/share/seclists/Discovery/SNMP/common-snmp-community-strings.txt target | tee 03-enum/snmp/onesixtyone_target.txt
snmpwalk -v2c -c public target 2>/dev/null | tee 03-enum/snmp/snmpwalk_public_target.txt
snmp-check target -c public | tee 03-enum/snmp/snmpcheck_target.txt
```
**Capture:** valid community strings, system info, interfaces, routes, running processes/software (version intel), user accounts where exposed.
**Safe stop:** read-only walks. Never write (`snmpset`) without explicit RoE approval.

### 4.5 DNS server — 53
```bash
# Authorized zone-transfer attempt against the exposed NS
dig AXFR @target client.tld | tee 03-enum/dns/axfr_target.txt
nmap -Pn -p53 --script dns-nsid,dns-recursion,dns-cache-snoop target -oN 03-enum/dns/nmap_dns_target.txt
```
**Capture:** AXFR exposure (full zone = high impact), open recursion, version (NSID).

### 4.6 VPN / IKE — 500/UDP, 4500/UDP
```bash
ike-scan -M target | tee 03-enum/vpn/ikescan_target.txt
ike-scan --aggressive -M -P03-enum/vpn/psk_target.txt target   # capture PSK hash if aggressive mode supported
```
**Capture:** IKE responder presence, aggressive-mode support (PSK hash exposure), vendor IDs.
**Safe stop:** capture handshake/PSK hash for offline reference; defer cracking decisions to RoE.

### 4.7 Databases — 1433/3306/5432/1521/27017/6379/etc.
```bash
nmap -Pn -sV --script "<db>-info" -p<port> target -oN 03-enum/db/nmap_db_target.txt
# Examples: ms-sql-info, mysql-info, oracle-tns-version (odat), mongodb-info
```
**Capture:** DB engine + version, instance names, whether auth is required (unauthenticated exposure = finding), TLS posture.
**Safe stop:** confirm reachability/auth requirement. No data dumping beyond schema/table-name proof.

### 4.8 Other common externals
| Service | Port | First-look enumeration |
|---------|------|------------------------|
| FTP | 21 | `nmap --script ftp-anon`; anonymous login check |
| SSH | 22 | banner/version, auth methods (`ssh -v`), key vs password |
| RDP | 3389 | `nmap --script rdp-ntlm-info`, NLA status, cert |
| WinRM | 5985/5986 | `netexec winrm target`; auth requirement |
| LDAP | 389/636 | `nmap --script ldap-rootdse`; anonymous bind |
| Kerberos | 88 | username validation via `impacket-GetNPUsers` (AS-REP, no-preauth) |

**Validation gate (exit Phase 3):** every in-scope service has an enumeration artifact in `03-enum/`; auth surfaces and version intel catalogued; candidate weaknesses queued for Phase 4.

## 5. Phase 4 — Vulnerability Identification & Validation

**Goal:** Turn enumeration into a ranked list of **candidate findings**, then confirm each one. Separate scanner output (candidate) from validated finding (evidence).

### 5.1 Automated candidate generation
```bash
# Template-based checks across discovered web/services
nuclei -l 03-enum/web/urls.txt -severity low,medium,high,critical \
  -o 04-vulnscan/nuclei.txt

# Map service versions to public exploits/advisories (reference, not auto-run)
searchsploit <product> <version>            # repeat per service:version from nmap -sV
# Authenticated/credentialed scan where provided: Nessus / OpenVAS (gvm) export → 04-vulnscan/
```

### 5.2 Manual validation loop (per candidate)
For each candidate, record:
1. **Affected component** (host:port, product, version).
2. **Entry point** (how it's reached externally).
3. **Confirmation method** (the exact request/command that proves it).
4. **Observed vs. expected** result.
5. **CWE** mapping (state "uncertain" if not confident — do not fabricate).
6. **CVSS** vector (only if supportable by observed facts; otherwise mark "to be scored").

> Do not promote a scanner hit to a finding without independent confirmation. If a candidate cannot be safely confirmed externally (e.g., would require destructive action), document it as **"unconfirmed — confirmation method X requires Y authorization."**

### 5.3 Decision tree
```
Candidate finding
   ├─ Confirmable non-destructively?  ── yes ─► validate, capture evidence ─► Phase 5 (if RoE allows exploitation)
   │                                   └─ no ─► document as unconfirmed + required method, request RoE expansion
   ├─ Requires credentials you hold (provided/found)? ─► authenticated validation
   └─ Auth weakness (spray candidate)? ─► Phase 5.2 spraying (lockout-aware)
```

**Validation gate:** each finding is either **Confirmed** (with evidence) or **Unconfirmed** (with the method needed). No "maybe" without a path.

## 6. Phase 5 — Exploitation / Initial Access (Bounded)

**Goal:** Demonstrate real-world impact with the **minimum** action necessary, only if RoE permits exploitation. Prefer benign PoCs proving reachability/impact over full compromise chains. MITRE ATT&CK: Initial Access (TA0001).

> Allowed only when Phase 0 RoE permits. Otherwise stop at Phase 4 and report validated vulnerabilities with reachability evidence.

### 6.1 Exploitation principles
- **Minimal proof:** demonstrate the primitive (e.g., command execution returns `id`/`whoami`, file read returns a known marker), then stop. No persistence, no credential theft beyond what proves impact, no lateral movement off the authorized host without explicit scope.
- **Reversibility:** avoid actions that alter or destroy data. If a write is required to prove impact, write a benign marker to an agreed location and remove it.
- **Logging:** capture request/response, timestamps, and target identifiers for every attempt (success and failure) into `06-access/`.

### 6.2 Credential attacks (lockout-aware)
```bash
# Password SPRAYING — many accounts, ONE password, then wait out the lockout window.
# Only on services without lockout, OR with confirmed lockout policy + safe spray count.
netexec smb target -u 01-osint/usernames.txt -p 'Season2025!' --continue-on-success \
  | tee 06-access/spray_smb_target.txt

# Targeted guessing only on accounts that CANNOT be locked out:
hydra -L users.txt -p 'KnownPassword!' target ssh -t 1 -o 06-access/hydra_ssh_target.txt
```
> **Lockout rule:** model the lockout policy first (from SMB/LDAP enumeration). Spray ≤ (threshold − 1) attempts per observation window, then wait. Track per-account attempt counts to avoid locking users out.

### 6.3 Service exploitation
```bash
# Use validated, version-matched exploits from Phase 4. Confirm target is vulnerable first.
searchsploit -m <id>            # stage locally, read it before running
# Run against the single confirmed target only; capture proof-of-execution output to 06-access/
```
For unauthenticated exposures (open DB, exposed admin panel, AXFR, open relay), the **proof is the access itself** — capture a screenshot/log showing the unauthenticated read and stop.

### 6.4 Safe stopping points by impact class
| Impact | Proof that is enough | Do NOT |
|--------|----------------------|--------|
| RCE | `id`/`hostname`/`whoami` output | Drop implants, persist, pivot |
| Auth bypass | Reach a privileged view/endpoint | Modify records, escalate further |
| Unauth data exposure | Schema/table names + row counts | Dump PII/sensitive contents |
| File read | Retrieve a benign known marker | Read secrets beyond proof |
| DoS-class | Theoretical analysis only | Trigger it against production |

### 6.5 Decision point
- Achieved minimal proof → record evidence, **stop**, return to validate next finding.
- Exploitation needs to chain off-host or off-scope → halt, document the chain hypothesis, request scope expansion in writing.

**Validation gate:** every exploited finding has reproducible steps + captured proof in `06-access/`, and the target is left in its original state (markers removed, no residual artifacts).

## 7. Phase 6 — Evidence Handling

**Goal:** Maintain an immutable, traceable evidence chain so every report claim links to raw data.

1. Store raw tool output in `07-evidence/raw/`; never edit raw files.
2. Screenshots: full-window, including target identifier + timestamp, in `07-evidence/screenshots/`.
3. Network captures (where used) in `07-evidence/pcaps/`.
4. Maintain an evidence index mapping `finding-id → evidence files → command run`.

```bash
# Hash raw evidence for integrity
find 07-evidence/raw -type f -exec sha256sum {} \; > 07-evidence/raw/SHA256SUMS.txt
```

**Validation gate:** each finding ID resolves to one or more hashed evidence artifacts and the exact command/request that produced it.

## 8. Phase 7 — Reporting

**Goal:** Produce a report clear enough for a Program Manager and technically defensible for engineers. Claims trace to evidence; impact is not overstated.

### 8.1 Report structure
1. Executive summary (business risk, themes, no jargon).
2. Scope, methodology, RoE, testing window, source IPs.
3. Findings (ordered by risk), using the template below.
4. Methodology/coverage statement (what was and wasn't tested).
5. Evidence appendix (cross-referenced to `07-evidence/`).
6. Remediation roadmap (prioritized).

### 8.2 Finding template
```markdown
### [Finding ID] <Title>

**Summary:** <1–2 sentence plain-language description and why it matters.>

**Affected component:** <host:port, product, version>
**Attack surface:** <External service / endpoint reached>
**Weakness classification:** <plain description>
**CWE mapping:** CWE-### <name>   (state "uncertain" if not confidently mappable)
**CVSS:** <vector string> → <Base score / severity>   (omit or mark "not scored" if unsupportable)

**Preconditions:** <What must be true for exploitation: network position, creds, config state.>

**Evidence:**
<Observed facts only — command output, screenshots, response excerpts.
Reference 07-evidence/ artifacts by filename + hash.>

**Root cause:** <Underlying defect: missing patch, misconfig, weak default, design flaw.>

**Impact:** <Concrete, evidence-bounded consequence. No speculation beyond what was shown.>

**Exploitability:** <How reachable/reliable, preconditions, skill/access required.
Separate CONFIRMED from INFERRED explicitly.>

**Reproduction steps:**
1. <Exact, repeatable steps/commands.>
2. ...

**Expected vs. observed result:**
- Expected (secure): <what a hardened system would do>
- Observed: <what this target did>

**Fix recommendation:** <Specific, actionable remediation. Patch version / config change / control.>

**Validation after fix:** <The same check that fails pre-fix and must pass post-fix.>

**Residual risk:** <What remains after remediation; compensating controls.>
```

### 8.3 Severity & quality checks before delivery
- Each finding: evidence present, CWE/CVSS supportable or explicitly marked uncertain, reproduction validated by a second pass.
- No invented CVEs, versions, paths, or exploitability claims.
- Executive summary readable without the technical sections.

**Validation gate:** report cross-references resolve; a peer can reproduce each finding from the steps alone.

## 9. Cross-Phase Decision Summary

```
Phase 0  Authorization complete? ─ no ─► STOP
         └ yes ─► scope normalized ─► Phase 1
Phase 1  In-scope live host:IP candidates ─► Phase 2
Phase 2  Open ports mapped to service+version ─► Phase 3
Phase 3  Per-service enumeration + auth surface ─► Phase 4
Phase 4  Finding Confirmed? ── yes + RoE allows exploit ─► Phase 5
         │                  └─ yes, exploit not allowed ─► report as validated vuln
         └─ Unconfirmed ─► document method needed, request RoE
Phase 5  Minimal proof captured ─► Phase 6 ─► Phase 7
```
