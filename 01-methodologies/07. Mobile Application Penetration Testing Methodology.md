# Mobile Application Penetration Testing Methodology

**Document type:** Repeatable assessment playbook (iOS + Android; static + dynamic + instrumentation)
**Scope of applicability:** Native, hybrid, and cross-platform (Flutter/React Native) mobile apps you are authorized to test, plus their on-device attack surface. The app's **backend** is in scope as a hand-off, not duplicated here.
**Relationship to the other playbooks:** A mobile app is a **thick client to a backend** — the highest-impact bugs are usually server-side. Test the on-device surface here; hand the backend off to the **API/Microservices** and **Web** methodologies (BOLA/BFLA, injection, auth at the data layer). Run every confirmed finding through the **Vulnerability Lifecycle** methodology (validation → safe PoC → impact → remediation → regression). Reuse the **Fuzzing** methodology for native-library (`.so`/Mach-O) parsing surfaces.
**Standards referenced:** OWASP **MASVS** (Mobile Application Security Verification Standard) + **MASTG** (Mobile Application Security Testing Guide) and the **MASWE** weakness enumeration; CWE, CVSS v3.1/v4.0, MITRE ATT&CK (Mobile), NIST SP 800-115. MASVS control groups: **STORAGE, CRYPTO, AUTH, NETWORK, PLATFORM, CODE, RESILIENCE, PRIVACY**.
**Toolchain assumed:** This Kali build — `apktool`, `dex2jar` (`d2j-*`), `jadx`/`jadx-gui`, `jd-gui`, `baksmali`/`smali` (libsmali-java), `bytecode-viewer`, Android SDK/`adb`/`fastboot`, `ghidra`, `radare2`/`rizin` (native libs), Burp/ZAP + `mitmproxy`, `nuclei`. Frida/`frida-tools`, Objection, MobSF, Drozer, `apkleaks`, `frida-ios-dump`, `class-dump`, Keychain-Dumper, SSL Kill Switch, `scrcpy`, an emulator/Genymotion referenced where relevant (install / device-side).

> **Operating constraints baked into this methodology**
> - **The app is a client; the server is where the breach is.** Always test the backend it talks to (handed to the API/Web playbooks). On-device findings (insecure storage, weak crypto, IPC) matter, but cross-tenant/BOLA/auth bypass usually live server-side.
> - **Test devices, not user devices.** Use dedicated rooted/jailbroken devices or instrumented (Frida-gadget-repackaged) builds you own. Never test against a real user's device or account.
> - **Instrumentation is required and must be authorized.** Frida/Objection (method hooking, pinning/root-detection bypass) is core to dynamic mobile testing; confirm it's permitted, and note that bypassing RESILIENCE controls is a *test of those controls*, not a finding-by-itself unless it exposes a downstream weakness.
> - **Non-destructive, account-scoped PoC.** Use test accounts and benign markers; no bulk PII, no destructive actions against the production backend. Honor app-store/provider terms.
> - Map findings to **MASVS/MASWE + CWE + ATT&CK(Mobile)**; separate evidence from hypothesis; each finding enters the Vulnerability Lifecycle.

## 0. Phase 0 — Scoping & Test-Environment Setup

**Goal:** Fix authorization, platforms, builds, devices, and backend hand-off before testing.

### 0.1 Authorization & scope — `00-scope/`
1. Signed authorization naming the app(s), platform(s) (iOS/Android), package/bundle IDs, and **backend** in/out of scope (API hosts).
2. **Build source:** store build vs a provided IPA/APK (+ source for white-box). Note version/build number/commit.
3. **Devices:** rooted Android / jailbroken iOS test devices, or non-jailbroken with Frida-gadget repackaging (Android) / sideload-provisioned builds (iOS). OS versions to cover.
4. **Backend hand-off:** confirm the API is authorized for testing (it usually needs its own RoE) — the mobile client is the map to that API.
5. **Accounts:** ≥2 test accounts per role (for authz testing at the backend), plus any multi-tenant/org accounts.
6. **RoE:** instrumentation allowed? production vs staging backend? real vs marker data? jailbreak/root permitted?
7. Privacy/data-handling constraints (the app may process PII).

### 0.2 Environment
```bash
# Android
adb devices                                  # confirm test device/emulator
adb install app.apk
# Proxy + CA for interception (configure Burp/mitmproxy; install user/system CA)
# Frida server on device (matching arch); Objection for the workflow
objection -g <package> explore               # (install) — REPL + common bypasses
# iOS (jailbroken)
frida-ps -U                                  # confirm Frida over USB
# Non-jailbroken: repackage with frida-gadget (Android) / re-sign (iOS) per RoE
```

**Validation gate:** authorization stored; platforms/builds/devices ready; backend hand-off confirmed; ≥2 test accounts; interception + instrumentation working.

## 2. Phase 1 — Recon & App Acquisition

**Goal:** Obtain the build, inventory its components, and extract the backend surface. MITRE ATT&CK (Mobile): Reconnaissance.

```bash
# Android: pull an installed APK if not provided
adb shell pm path <package>                  # locate base.apk + splits
adb pull <path>/base.apk 01-recon/
# Extract endpoints, secrets, deeplinks from the package
apkleaks -f base.apk -o 01-recon/apkleaks.txt   # (install) URLs, keys, endpoints
# iOS: obtain decrypted IPA from a jailbroken device (frida-ios-dump) per RoE
```
Record: app/bundle version, declared permissions, embedded **backend endpoints** (→ `10-backend/` inventory for the API hand-off), third-party SDKs, and store-listing metadata.

**Validation gate:** build acquired; endpoint + SDK inventory captured; backend surface staged for the API/Web hand-off.

---

## 3. Phase 2 — Static Analysis (MASVS-CODE / STORAGE / CRYPTO)

**Goal:** Decompile and review the app without running it. Identify secrets, weak crypto, dangerous configs, and the IPC surface to test dynamically.

### 3.1 Decompile & decode
```bash
apktool d base.apk -o 02-static/decompiled/apktool        # manifest + smali + resources
jadx -d 02-static/decompiled/jadx base.apk                # Java/Kotlin source
# (or) d2j-dex2jar base.apk -o classes.jar ; jd-gui classes.jar
# iOS: class-dump / Ghidra on the Mach-O; Info.plist + entitlements review
```

### 3.2 Manifest / Info.plist review (high-yield)
- **Android (`AndroidManifest.xml`):** `android:debuggable="true"`, `android:allowBackup="true"`, **exported** components (`activity`/`service`/`receiver`/`provider` with `exported="true"` or implicit intent-filters), custom permissions, `usesCleartextTraffic`, **Network Security Config** (trust anchors, pinning), deeplink/intent-filter schemes, `minSdkVersion`.
- **iOS (`Info.plist`/entitlements):** `NSAppTransportSecurity` exceptions (ATS), URL schemes, associated domains (Universal Links), background modes, keychain access groups, `UIFileSharingEnabled`.

### 3.3 Secrets & dangerous code
```bash
# Hardcoded keys/tokens/URLs in decompiled code + resources
grep -rE '(api[_-]?key|secret|password|token|aws_|BEGIN .*PRIVATE)' 02-static/decompiled/ > 02-static/secrets/grep.txt
# MobSF for automated static (manifest, secrets, perms, crypto, trackers)
# (install) mobsf — upload APK/IPA for a baseline static report
```
Review crypto usage (algorithms, hardcoded keys/IVs, ECB, custom crypto), logging of sensitive data, WebView config (`setJavaScriptEnabled`, `addJavascriptInterface`, `setAllowFileAccess`), and dynamic-code loading.

### 3.4 Native libraries & SDKs
- Native `.so` (Android) / embedded frameworks (iOS): triage with Ghidra/radare2 for parsing surfaces (→ Fuzzing methodology if a custom parser handles untrusted input).
- Third-party SDK inventory + versions → SCA for known-vulnerable components (MASVS-CODE).

**Validation gate:** decompiled source + manifest/plist reviewed; secrets/crypto/WebView/native/SDK findings captured (as candidates → Lifecycle Phase 2); exported-component + deeplink list staged for Phase 7.

## 4. Phase 3 — Local Data Storage (MASVS-STORAGE)

**Goal:** Find sensitive data stored insecurely on-device. *The most common mobile finding class.*

Inspect (Android paths under `/data/data/<package>/`; iOS under the app sandbox):
```bash
# Android (root): pull and inspect app data
adb shell run-as <package> ls -la shared_prefs databases files
adb shell run-as <package> cat shared_prefs/*.xml          # tokens/PII in prefs?
adb pull /data/data/<package>/databases 03-storage/        # sqlite — open with sqlitebrowser
# Logs / external storage / clipboard
adb logcat | grep -iE 'token|password|secret'              # sensitive data in logs
adb shell ls -la /sdcard/Android/data/<package>            # world-accessible files?
```
Checklist: SharedPreferences/`NSUserDefaults`/plist storing secrets; SQLite/Realm/Core Data unencrypted PII; **Keystore/Keychain** misuse (data outside hardware-backed storage); cached files; **backups** (`allowBackup`, iTunes/iCloud); **clipboard** exposure of sensitive fields; **screenshots** on backgrounding (sensitive data in the app-switcher snapshot); keyboard cache; WebView cache/cookies.

**Validation gate:** every storage location enumerated; sensitive data at rest identified with the file/key path as evidence; secure-storage gaps recorded.

---

## 5. Phase 4 — Cryptography (MASVS-CRYPTO)

**Goal:** Assess key management and cryptographic use.
- Hardcoded keys/IVs (Phase 3.3), weak/deprecated algorithms (DES/3DES/RC4/MD5/SHA1 for security), ECB mode, predictable IV/nonce, insecure random.
- Key storage: are keys in the **hardware-backed** Keystore/Keychain/Secure Enclave, or recoverable from storage/code?
- Custom crypto / rolling-your-own; improper certificate/key validation.

**Validation gate:** crypto findings tied to code locations; key-management weaknesses demonstrated (e.g., key recoverable from storage).

---

## 6. Phase 5 — Authentication & Session (MASVS-AUTH)

**Goal:** Test on-device auth and how the app handles credentials/tokens/sessions. *Server-side auth enforcement is tested in the backend hand-off (Phase 10).*
- **Local authentication bypass:** biometric/PIN gating that protects access without server-side enforcement — bypass via Frida (hook the success callback) to confirm it's not security-relevant on its own.
- **Token/credential storage:** access tokens/refresh tokens/passwords in prefs/plist/keychain (Phase 3) — scope, protection, and whether they survive logout.
- **Session:** token lifetime, invalidation on logout, binding; OAuth/OIDC flows in the app (redirect handling, PKCE, deeplink token interception).

```bash
objection -g <package> explore
# android keystore list ; ios keychain dump ; hook auth methods to test local-auth gating
```

**Validation gate:** local-auth gating tested (bypass demonstrates it's client-side only); token storage/session assessed; OAuth deeplink handling reviewed.

---

## 7. Phase 6 — Network Communication (MASVS-NETWORK)

**Goal:** Verify transport security and intercept traffic to reach the backend. *Interception unlocks the API hand-off.*

### 7.1 TLS & pinning
- Confirm TLS everywhere; no cleartext (`usesCleartextTraffic`/ATS exceptions). 
- **Certificate pinning:** attempt interception with your CA; if blocked, bypass for *testing* (Objection `android sslpinning disable` / iOS equivalent, or a Frida pinning-bypass script, or SSL Kill Switch on jailbroken iOS). A bypassable pin is expected for testing; **weak or absent pinning** on a high-value app is the finding.
```bash
objection -g <package> explore -s "android sslpinning disable"
# then route device traffic through Burp/mitmproxy and capture
```

### 7.2 Traffic analysis → backend hand-off
With interception working, capture the full API conversation. This **is** the backend attack-surface inventory:
- Endpoints, auth headers/tokens, object references (→ BOLA testing), parameters, and data exposure.
- Hand this to the **API/Microservices** and **Web** methodologies for BOLA/BFLA, injection, auth-at-the-data-layer, and tenant isolation (the high-impact work).

**Validation gate:** TLS/pinning posture recorded; traffic intercepted and the API surface captured to `10-backend/` for the hand-off.

---

## 8. Phase 7 — Platform Interaction & IPC (MASVS-PLATFORM)

**Goal:** Test inter-process and platform attack surfaces — the on-device "remote" attack surface other apps can reach.

### 8.1 Android exported components & intents
```bash
# Drozer (install): enumerate + attack exported components
drozer console connect
#   run app.package.attacksurface <package>
#   run app.activity.start / app.provider.query / app.service.start ...
# Manual: invoke exported components via adb
adb shell am start -n <package>/<exported-activity>        # reach screens without auth?
adb shell content query --uri content://<exported-provider>/...   # data leak / SQLi in provider?
```
- **Exported activities:** reach authenticated screens, bypass login.
- **Content providers:** unauthorized data access, **SQL injection** in provider queries, path traversal in file-backed providers.
- **Broadcast receivers / services:** injectable actions, privilege issues.
- **Deeplinks / intent filters:** parameter injection, open redirect, auth bypass via crafted intents; **WebView** `addJavascriptInterface` / file-access → local file theft / RCE-in-WebView.

### 8.2 iOS platform
- **Custom URL schemes / Universal Links:** parameter injection, action invocation without auth, hijacking (scheme squatting).
- **Pasteboard** (sensitive data), keyboard cache, backgrounding screenshot, `UIWebView`/`WKWebView` config, app extensions, inter-app data sharing.

**Validation gate:** every exported/IPC surface enumerated and tested; component-level access/injection demonstrated with the exact intent/URI; WebView config assessed.

## 9. Phase 8 — Dynamic Instrumentation & Runtime (MASVS-AUTH/PLATFORM/RESILIENCE)

**Goal:** Manipulate the running app to test client-side controls and business logic. MITRE ATT&CK (Mobile): runtime manipulation.

```bash
objection -g <package> explore
# Examples:
#   memory search / dump ; hook a method to bypass a client-side check;
#   force a feature flag / paid-feature gate; tamper with a response handler.
frida -U -f <package> -l hook.js --no-pause   # custom Frida hooks
```
- **Client-side control bypass:** any security decision made on-device (jailbreak/root gate, feature/paywall gate, validation) can be hooked — confirm it's enforced server-side, not just in the client.
- **Runtime data:** extract secrets/keys/tokens from memory; observe decrypted data.
- **Business-logic:** manipulate in-app purchase validation, promo/coupon logic, limits — confirm server-side enforcement.

**Validation gate:** client-side controls tested via hooking; any control enforced only on-device flagged (and re-verified against the backend in Phase 10).

---

## 10. Phase 9 — Resilience & Anti-Tampering (MASVS-RESILIENCE)

**Goal:** Assess the strength of reverse-engineering and tampering defenses. *These are defense-in-depth; weakness here is reportable but rarely the primary risk — context matters (e.g., high-value finance/DRM apps).*
- **Root/jailbreak detection:** present? Bypassable (Objection/Frida)? 
- **Anti-debugging / anti-hooking / anti-tamper / integrity checks:** present and effective?
- **Obfuscation:** is sensitive logic/strings obfuscated, or trivially readable (Phase 2)?
- **SSL pinning** strength (Phase 6).

Report the *posture* (which controls exist, how easily bypassed) calibrated to the app's risk profile — don't overstate a missing-root-detection finding on a low-sensitivity app.

**Validation gate:** RESILIENCE controls inventoried with bypass difficulty; severity calibrated to the app's risk profile.

---

## 11. Phase 10 — Backend & API (Hand-off)

**Goal:** Test the server — usually where the real risk is. Use the captured API surface (Phase 6) and the **API/Microservices** + **Web** methodologies.
- **BOLA/IDOR:** the object references seen in mobile traffic, replayed across your ≥2 test accounts (cross-user/cross-tenant).
- **BFLA / authz:** mobile-only or hidden endpoints reachable from a lower-privilege account; server-side enforcement vs client gating.
- **Injection, mass assignment, excessive data exposure, SSRF** per the API/Web playbooks.
- **Auth/session** at the server: token validation, JWT issues, OAuth.

> The mobile client frequently exposes endpoints the web UI hides. The most impactful mobile-engagement findings are typically these server-side issues — do not stop at on-device findings.

**Validation gate:** backend tested via the API/Web methodologies using the mobile-derived surface; authz re-tested with multiple accounts.

---

## 12. Phase 11 — Privacy & Data Collection (MASVS-PRIVACY)

**Goal:** Assess data handling, permissions, and third-party tracking.
- **Permissions:** over-broad/unnecessary permissions vs declared functionality.
- **Tracking SDKs / data leakage:** PII/identifiers sent to third parties (observed in Phase 6 traffic + SDK inventory from Phase 2).
- **Sensitive data exposure:** in logs, clipboard, backups, screenshots, analytics.

**Validation gate:** permission set + tracker inventory + data-flow leakage documented.

---

## 13. Phase 12 — Exploitation, Impact & Reporting

**Goal:** Deliver evidence-based findings; each runs through the **Vulnerability Lifecycle** (validation → safe PoC → impact → remediation → regression).

### 13.1 Safe-PoC discipline (mobile)
- Storage/crypto: show the secret/PII at its file/key path; don't exfiltrate beyond proof.
- IPC: invoke the component with a benign payload showing the access/injection; no destructive provider writes.
- Client-bypass: hook to demonstrate the control is client-side; then prove (or disprove) server-side enforcement against a test account.
- Backend: per the API/Web safe-PoC rules (own test accounts, markers, no bulk PII).

### 13.2 Report structure
1. Executive summary (on-device risk + backend risk; key chains).
2. Scope, platforms, builds, devices, backend hand-off.
3. Findings (ranked), template below.
4. Coverage statement (MASVS control groups covered, per platform).
5. Evidence appendix; remediation roadmap (secure storage, pinning, server-side enforcement, IPC hardening).

### 13.3 Finding template
```markdown
### [Finding ID] <Title>
**Summary:** <plain-language risk for a PM>
**Platform / component:** <iOS/Android; app component / endpoint>
**Attack surface / entry point:** <static config / IPC / traffic / runtime / backend>
**Weakness classification:** <e.g., insecure storage, missing pinning, exported provider SQLi, client-side auth, BOLA (backend)>
**MASVS control:** MASVS-<GROUP>     **MASWE/CWE:** MASWE-### / CWE-### (uncertain?)
**MITRE ATT&CK (Mobile):** <Txxxx>
**CVSS:** <vector> → <score/severity>  (or "not scored" + why)
**Preconditions:** <device state, auth, position; instrumentation needed?>
**Evidence:** <observed: file/key path, intent/URI, captured request, hook output, screenshots + sha256>
**Root cause:** <on-device misconfig / missing server-side enforcement / weak crypto, etc.>
**Impact:** <evidence-bounded; CONFIRMED vs INFERRED; note if backend-amplified>
**Exploitability:** <reliability, preconditions, device requirements; CONFIRMED vs INFERRED>
**Reproduction steps:** <exact, repeatable; tool + command>
**Expected vs. observed:** <secure behavior vs what the app did>
**Fix recommendation:** <hardware-backed storage, strong pinning, server-side authz, IPC export hardening, remove secrets>
**Validation after fix:** <same check succeeds pre-fix, fails post-fix>
**Residual risk:** <related surfaces; compensating controls>
```

### 13.4 Quality checks
Each finding: evidence present; MASVS/MASWE/CWE supportable or marked uncertain; reproduction validated; client-side vs server-side enforcement distinguished; backend findings cross-referenced to the API/Web work; exec summary standalone.

**Validation gate:** every finding links to hashed evidence; MASVS coverage stated per platform; client-vs-server impact distinguished; each finding closed through the Lifecycle methodology.

## 17. Appendix C — Safe-PoC Guidance (mobile)
- **Storage/crypto:** reveal the secret/PII at its path/key; no exfiltration beyond proof.
- **IPC/provider:** invoke with a benign payload; demonstrate SQLi with a non-destructive query; no provider writes.
- **Client-side bypass:** hook to show the control is client-only; then verify server-side enforcement against a *test account*.
- **Pinning:** bypass for testing is expected; the finding is weak/absent pinning, not the bypass.
- **Backend:** per API/Web safe-PoC — test accounts, markers, no bulk PII, no destructive actions.
- **Devices/accounts:** owned test devices and accounts only; never a real user's device or data.

---

*This methodology treats the mobile app as a thick client: test the on-device surface fully against OWASP MASVS, but route the backend to the API/Web playbooks where the high-impact authz/injection findings usually live, and run every finding through the Vulnerability Lifecycle. Distinguish client-side from server-side enforcement, use owned test devices/accounts with non-destructive PoCs, and map findings to MASVS/MASWE + CWE + ATT&CK(Mobile). Mark mapping and exploitability uncertainty explicitly.*
