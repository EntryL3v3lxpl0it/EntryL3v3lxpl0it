# SaaS & Cloud-Exposed Application Testing Methodology

**Document type:** Repeatable assessment playbook (multi-tenant SaaS + the cloud substrate it runs on)
**Scope of applicability:** Multi-tenant SaaS applications and cloud-hosted applications exposed to the internet — assessed as the **vendor** (full architecture incl. cloud substrate) or as a **customer/third party** (your own tenant + your connected cloud only)
**Relationship to other playbooks:** Reuse the **Web** methodology (WSTG classes), the **API/Microservices** methodology (OWASP API Top 10, SSRF-as-pivot, gateway/inter-service trust), and the **AD/Windows-Enterprise** methodology where hybrid Entra is in play. **This** document adds what is unique to SaaS/cloud-exposed apps: multi-tenant isolation, cloud identity/federation, the cloud substrate (IMDS/IAM/storage), subdomain takeover, and SaaS business logic.
**Standards referenced:** OWASP WSTG / API Security Top 10 / ASVS, CSA Cloud Controls Matrix, MITRE ATT&CK (Enterprise + Cloud/IaaS), CWE, CVSS v3.1/v4.0, NIST SP 800-115/800-204; the **cloud shared-responsibility model** and each provider's penetration-testing/acceptable-use policy.
**Toolchain assumed:** This Kali build — Burp/ZAP/Caido + `mitmproxy`, `subfinder`/`amass`/`dnsx` (asset discovery), `subjack` (subdomain takeover), `cloud-enum`/`cloudbrute`/`s3scanner` (cloud assets/storage), `pacu` (AWS), `azurehound` (Entra/Azure), `nuclei` (takeover/exposure/cloud templates), `trufflehog`/`gitleaks` (secrets), `trivy`/`syft`/`cosign` (IaC/images), `ffuf`/`arjun`, `hashcat -m 16500` (JWT). Provider CLIs (`aws`/`az`/`gcloud`), `ScoutSuite`/`Prowler`/`cloudsplaining`, `ROADtools`/`AADInternals`, `jwt_tool`, SAML Raider, `interactsh`/Burp Collaborator referenced where relevant (install / Burp Pro).

> **Operating constraints baked into this methodology**
> - **Authorization here is uniquely fraught.** SaaS sits on infrastructure you usually don't own. Before any active test you must establish: (a) who authorizes — the SaaS vendor and/or the customer; (b) the **shared-responsibility line** (what's the provider's vs the customer's vs the SaaS vendor's); and (c) the **cloud provider's penetration-testing policy** (AWS/Azure/GCP each publish one; some require notification, all prohibit attacking the provider's own infrastructure and other tenants).
> - **Never affect another tenant or the provider's infrastructure.** Tenant-isolation testing is done with **your own ≥2 provisioned test tenants**. Cross-tenant proof uses *your* tenants only. No DoS/resource-exhaustion against shared platform components.
> - **Tenant isolation is the marquee concern.** The single highest-value SaaS question is: can tenant A reach tenant B's data, config, or compute? Drive testing from that question.
> - **SSRF is the cloud keystone.** In cloud-hosted apps, SSRF reaching the instance metadata service (IMDS) often yields cloud credentials → IAM → potentially cross-tenant. Treat every URL-consuming field as a path to the cloud control plane.
> - **Non-destructive, tenant-scoped PoC.** Prove impact with the minimum action, against your own tenants/markers; no bulk PII, no live cloud-resource destruction. Map findings to **OWASP + CWE + ATT&CK(Cloud)** and attribute each to the **shared-responsibility owner**.

## 0. Phase 0 — Scoping, Perspective & Shared Responsibility

**Goal:** Establish who authorizes, from what perspective, and within which provider policy — the make-or-break phase for this domain.

### 0.1 Perspective (pick the model) — `00-scope/`
- **Vendor assessment:** you test the SaaS as its operator — full application + cloud architecture, IAM, storage, serverless, the whole substrate.
- **Customer/third-party assessment:** you test a SaaS you *consume* — limited to **your own tenant** and **your own connected cloud (BYOC)**. You may NOT test the vendor's shared infrastructure or other tenants, and the vendor's authorization (or a contractual pentest clause) is typically required.

### 0.2 Authorization & policy (hard blocker)
1. Signed authorization from the correct party (vendor for vendor-assessment; vendor + customer for third-party where the SaaS ToS requires it).
2. **Shared-responsibility map:** document, per component, who owns security — cloud provider (infra), SaaS vendor (platform/app), customer (config/data/identity). Findings get attributed to the right owner.
3. **Cloud-provider testing policy:** confirm and honor AWS/Azure/GCP pentest/acceptable-use policy (no provider-infra attacks, no other-tenant impact, notification where required). Record source IPs.
4. **Tenancy:** obtain/provision **≥2 isolated test tenants** (for isolation testing) and accounts for **every role** within a tenant (org/super admin, workspace/customer admin, member, anonymous).
5. **Environment:** prod vs staging; data residency/PII constraints; whether you may create/modify/delete test data.
6. **In-scope cloud identities/accounts** (vendor-assessment): which accounts/subscriptions/projects and which IAM principals.
7. **RoE intensity:** automated scanning against prod? rate-limit/resource testing (availability-sensitive on shared infra — usually restricted)? real vs marker data for exfil proof?
8. Emergency stop + vendor/provider/SRE contacts.

### 0.3 Decision point
- Third-party perspective → **hard boundary**: only your tenant + your connected cloud; any cross-tenant proof must use *your second tenant*; do not touch vendor infra.
- No provider authorization for infra-level testing → cap at application/identity layer; treat cloud-substrate issues as analysis/observation only.

**Validation gate:** perspective chosen; authorization from correct party; shared-responsibility map drafted; provider policy confirmed; ≥2 test tenants + role accounts provisioned.

## 2. Phase 1 — Cloud Asset Discovery & Attack-Surface Mapping

**Goal:** Enumerate the internet-exposed cloud footprint and the SaaS's tenant model. MITRE ATT&CK: Reconnaissance (TA0043).

### 2.1 Asset discovery pipeline
```bash
subfinder -d target.tld -all -silent | tee 01-discovery/assets/subs.txt
amass enum -passive -d target.tld -o 01-discovery/assets/amass.txt
dnsx -l 01-discovery/assets/subs.txt -a -cname -resp -silent | tee 01-discovery/dns/resolved.txt   # CNAMEs reveal cloud services
httpx -l 01-discovery/dns/resolved.txt -title -status-code -tech-detect -cdn -o 01-discovery/fingerprint/httpx.txt
```
CNAME/IP analysis reveals the cloud provider(s), CDN/edge, and managed services. Record provider per asset (drives substrate testing).

### 2.2 Object storage & cloud resources
```bash
cloud_enum -k <company> -k <company>-prod -k <company>-dev          # AWS/Azure/GCP storage + services
s3scanner scan -bucket-file 01-discovery/storage/candidates.txt     # public/listable/writable buckets
cloudbrute -d target.tld -k <company> -m storage                    # storage brute (AWS/Azure/GCP)
```
Note public/listable/writable buckets, exposed blobs, and predictable naming (often per-tenant: `tenant-<id>-uploads`).

### 2.3 Tenant model & SaaS surface
- Determine the **tenancy pattern:** subdomain-per-tenant (`acme.app.com`), path-per-tenant (`/t/acme`), or header/claim-based. This shapes isolation testing.
- Map the SSO/login endpoints, signup/onboarding, admin console, public API base, OpenAPI/GraphQL schema, marketplace/integration pages.
- Tenant enumeration: can valid tenant IDs/slugs be enumerated (signup checks, login responses, public profile URLs)?

**Validation gate:** exposed-asset inventory with provider attribution; storage exposure noted; tenancy pattern + SSO/API surface mapped; tenant-enumeration vector recorded.

## 3. Phase 2 — Identity, SSO & Access Management

**Goal:** Test how identity is established, federated, and provisioned. OWASP API2. MITRE ATT&CK: T1078.004 (Cloud Accounts), T1556.

### 3.1 SSO — SAML / OIDC
- **SAML:** signature validation (unsigned/stripped/wrapped assertions — XSW), recipient/audience/`NotOnOrAfter` enforcement, IdP-initiated abuse, replay; test with SAML Raider (Burp).
- **OIDC/OAuth:** `redirect_uri` validation, `state`/nonce, PKCE, token leakage, scope creep, `id_token` claim trust (email/`hd`/`tid`), account linking (login-with-Google takeover via unverified email).

### 3.2 Provisioning & onboarding
- **SCIM / JIT provisioning:** can attributes (role, group, tenant) be influenced during provisioning? Mass-assignment at signup?
- **Invitations / onboarding:** invite-acceptance binding, predictable invite tokens, role granted on accept, email-verification bypass, self-signup to an existing tenant's domain.

### 3.3 OAuth app integrations & consent
- Malicious/over-scoped app consent, dangerous default scopes, consent phishing surface, token storage; (Entra) `azurehound`/Graph permission review.

### 3.4 Tokens & sessions
- API tokens / personal access tokens (PATs): scope, entropy, revocation, tenant binding. JWT: `alg`/key confusion, claim-based authz (`tenant`/`role`/`org`) trusted blindly, weak HMAC (`hashcat -m 16500`).
- Session: lifetime, invalidation on logout/SSO-logout, fixation, cross-tenant session reuse.

**Validation gate:** SSO/federation, provisioning, consent, and token/session tested; any claim-based tenant/role trust flagged for Phase 3/4.

## 4. Phase 3 — Tenant Isolation (Marquee)

**Goal:** Prove (or disprove) that one tenant cannot reach another tenant's data/config/compute. **Use your own ≥2 test tenants.** OWASP API1. MITRE ATT&CK: T1530 (Data from Cloud Storage), T1213.

### 4.1 Cross-tenant object access
From Tenant A's session, substitute Tenant B's identifiers (object IDs, tenant ID/slug, resource URLs, file keys) and confirm whether A can read/modify B's objects. Iterate across every object reference and every API endpoint.
```
# Pattern: capture request as Tenant A user, replay swapping B's IDs/tenant context.
# Burp Autorize/Auth Analyzer with two tenant sessions automates the diff.
```

### 4.2 Tenant-context tampering
- Tenant selector in header/cookie/JWT claim/path: change it to B's and see if the backend re-validates membership or trusts it.
- Sub-tenant / child-org / workspace boundaries: can a workspace member reach a sibling workspace?

### 4.3 Shared-resource leakage
- Object storage: tenant A's signed URLs / bucket paths reaching B's objects; predictable per-tenant keys.
- Caches/search indexes/exports/reports/notifications: do they leak cross-tenant rows?
- Shared compute/DB: identifiers from B appearing in A's responses (excessive data exposure across the tenant boundary).

### 4.4 Tenant enumeration & admin-scope confusion
- Enumerate tenants (slugs/IDs); confirm whether a "global" admin scope is reachable from a tenant admin; test impersonation features for cross-tenant reach.

**Validation gate:** each isolation test demonstrated with two of *your* tenants; cross-tenant access (if any) captured with the request + which tenants; no real customer tenant touched.

## 5. Phase 4 — Authorization Within a Tenant

**Goal:** Test the role hierarchy and access control inside a tenant. OWASP API1/API3/API5. *Map from the role matrix (super/org admin → workspace/customer admin → member → anonymous).*

- **BOLA/IDOR** within tenant (cross-user, same tenant).
- **BFLA / vertical privilege escalation:** member reaching admin functions; force-browse admin/billing/SSO-config endpoints; server-side enforcement vs UI hiding.
- **Mass assignment / property-level (BOPLA):** set `role`/`isAdmin`/`tenantId`/`plan` on update; excessive data exposure in responses.
- **Impersonation features:** "log in as user" / support-impersonation — scope, audit, and whether it crosses tenant or role boundaries.

**Validation gate:** intra-tenant authz tested with ≥2 identities per role; response diffs captured; impersonation features assessed for scope and logging.

---

## 6. Phase 5 — Application Input & Injection

**Goal:** Cover the web/API injection surface (the keystone here is SSRF → cloud). Reference the Web and API methodologies for full class coverage; this phase focuses on what feeds Phase 6.

- Standard classes per the Web methodology: SQLi/NoSQLi, command injection, SSTI, XSS (S/R/DOM), XXE, file inclusion/upload, CRLF, deserialization, prototype pollution. Non-destructive PoC.
- **SSRF discovery (cloud keystone):** identify every URL/host-consuming field (webhooks, importers, image/PDF/preview, integration callbacks, "fetch from URL", SAML/OIDC metadata URLs). Confirm via OOB callback. This is the entry to Phase 6.

**Validation gate:** injection confirmed (not scanner-only); all SSRF-capable inputs catalogued for cloud-substrate pivoting.

## 7. Phase 6 — Cloud Substrate Exposure

**Goal:** From the application, reach the cloud control plane. MITRE ATT&CK: T1552.005 (Cloud Instance Metadata API), T1078.004, T1098.001. **Vendor-perspective, or customer's own connected cloud only; honor provider policy.**

### 7.1 SSRF → instance metadata (IMDS) → cloud creds
From a confirmed SSRF (5), attempt the metadata endpoint:
- AWS: `http://169.254.169.254/latest/meta-data/iam/security-credentials/<role>` (note IMDSv2 token requirement — a missing-IMDSv2 finding is itself reportable).
- Azure: `http://169.254.169.254/metadata/identity/oauth2/token` (requires `Metadata: true` header).
- GCP: `http://metadata.google.internal/computeMetadata/v1/` (requires `Metadata-Flavor: Google`).
Capture the credential/token as proof; **do not** abuse beyond demonstrating retrieval unless RoE explicitly permits IAM follow-on.

### 7.2 IAM analysis & privilege escalation (vendor / own cloud)
```bash
# With obtained or provided cloud credentials, in an AUTHORIZED account only:
pacu                                  # AWS enumeration + privesc paths
# ScoutSuite / Prowler / cloudsplaining (install) for IAM/posture review
azurehound -u <user> -p '<pass>' list # Entra/Azure graph (hybrid)
```
- Over-permissive roles, `iam:PassRole`/`sts:AssumeRole` chains, wildcards, privilege-escalation paths; whether a tenant-scoped workload identity can reach cross-tenant resources.

### 7.3 Object storage & secrets
```bash
s3scanner scan -bucket <name>          # listable/writable; per-tenant bucket reach
trufflehog filesystem <export>         # secrets in exports/configs/JS
gitleaks detect -s <repo>              # secrets in connected repos (vendor)
```
- Public/writable buckets, predictable per-tenant paths, secrets in client bundles/responses/IaC, exposed `.env`/config, KMS/CMK handling.

### 7.4 Serverless & managed data
- Serverless functions: event-injection, over-permissioned execution roles, exposed function URLs.
- Managed DB/datalake/LLM endpoints reachable from the app or via SSRF; auth-required check.

**Validation gate:** cloud-substrate findings proven with bounded read-only evidence (credential retrieved, bucket listed, role enumerated); no destructive cloud actions; all within authorized accounts and provider policy.

## 8. Phase 7 — SaaS Business Logic

**Goal:** Test the logic unique to multi-tenant SaaS commerce and lifecycle. OWASP API6. MITRE ATT&CK: T1496 (Resource Hijacking — careful/RoE).

- **Plan / feature-gating bypass:** access paid/enterprise features on a free/trial plan (client-side gating, missing server-side entitlement checks, direct API calls to gated endpoints).
- **Quota / billing / metering abuse:** bypass usage limits, manipulate metered counters, seat/license over-allocation, trial-reset/abuse.
- **Provisioning & lifecycle:** tenant creation/deletion flows, downgrade/upgrade race conditions, orphaned-resource access after offboarding.
- **Data export / deletion:** does deletion actually remove data (and across replicas/backups)? Does export leak more than the tenant should access? Right-to-be-forgotten integrity.
- **Audit-log integrity:** can a tenant admin tamper with or disable audit logs; are security-relevant actions logged?

**Validation gate:** logic findings reproduced with the exact request sequence; impact tied to a business consequence; tested within your tenants.

---

## 9. Phase 8 — Integrations & Supply Chain

**Goal:** Test the trust the SaaS places in third parties and connected systems.

- **Third-party app / marketplace integrations:** OAuth scope abuse, token storage, integration acting cross-tenant, malicious-app installation surface.
- **Webhooks:** SSRF via webhook target URL, signature/HMAC validation bypass, replay, webhook-to-internal pivot.
- **BYOC / connected cloud (customer-managed):** cross-account role trust (`ExternalId` enforcement), over-broad permissions granted to the SaaS, confused-deputy via the SaaS's role.
- **CI/CD & IaC** (vendor): pipeline secrets, unsigned artifacts, IaC misconfig (`trivy config`), dependency/supply-chain risk.

**Validation gate:** integration trust boundaries tested; webhook/BYOC trust validated; cross-account confused-deputy assessed where applicable.

## 10. Phase 9 — Subdomain & DNS Takeover

**Goal:** Find dangling DNS pointing at reclaimable cloud resources. MITRE ATT&CK: T1584.001 (Domains). *Common where SaaS provisions per-tenant or per-feature cloud resources.*

```bash
subjack -w 01-discovery/assets/subs.txt -t 50 -ssl -o 09-takeover/subjack.txt
nuclei -l 01-discovery/dns/resolved.txt -t http/takeovers -o 09-takeover/nuclei_takeover.txt
```
- CNAMEs pointing to de-provisioned cloud services (storage buckets, app platforms, CDNs) you could reclaim. **Prove with a benign marker on a resource you legitimately register; do not serve content impersonating the target.** Where reclaiming would require registering the resource, document the dangling record + the reclaimable service as the finding rather than taking it over.

**Validation gate:** takeover candidates verified to the safe limit (dangling record + reclaimable service identified); no impersonating content served.

## 11. Phase 10 — Exploitation & Chaining

**Goal:** Demonstrate real impact with minimal action; the SaaS-defining chains cross the tenant or cloud boundary.

### 11.1 Principles
- Minimal, tenant-scoped proof: cross-tenant → reach *your second tenant's* object; IMDS → retrieve the credential (no further cloud abuse unless RoE); storage → list a benign object; feature-gate → invoke one gated endpoint. No bulk PII, no live cloud destruction, no other-tenant impact.
- **Signature chains** (the deliverable): *SSRF → IMDS → cloud credentials → IAM enumeration → cross-tenant resource* (the SaaS-killer); *tenant-ID tampering → cross-tenant data*; *SAML signature bypass → admin of another org*; *subdomain takeover → SSO/cookie scope abuse*; *over-scoped OAuth integration → cross-tenant action*.

### 11.2 Capture
Per chain: each request/response, the boundary crossed (tenant? cloud? provider?), and a narrative annotated with the shared-responsibility owner. Store in `10-exploit/`.

**Validation gate:** each chain reproducible with captured proof, scoped to your tenants/authorized cloud; no provider-infra or other-tenant effect.

## 12. Phase 11 — Reporting

**Goal:** Evidence-based findings with tenancy impact and shared-responsibility attribution, readable by a PM and defensible to engineers.

### 12.1 Structure
1. Executive summary (can a tenant breach isolation or reach the cloud control plane? key chains, posture).
2. Scope, **perspective** (vendor vs customer), shared-responsibility map, provider-policy adherence, tenants used.
3. Findings (ranked), template below.
4. Notable chains (narrative + boundary crossed + impact).
5. Coverage statement (tenancy, identity, cloud substrate, integrations, OWASP categories).
6. Evidence appendix; remediation roadmap (isolation enforcement at the data layer, IMDSv2, IAM least privilege, SSO hardening, entitlement checks, secret management).

### 12.3 Quality checks
Each finding: evidence present (with which test tenants); OWASP/CWE/ATT&CK + tenancy impact + responsibility owner populated or marked uncertain; reproduction validated; cross-tenant proof uses only your tenants; no provider-infra/other-tenant effect; exec summary standalone.

**Validation gate:** every finding links to hashed evidence and a responsibility owner; tenancy impact stated; chains documented; provider-policy adherence affirmed.

## 17. Appendix D — Safe-PoC Guidance
- **Cross-tenant:** reach an object in *your second tenant*; non-sensitive field as proof. Never another customer's data.
- **SSRF→IMDS:** retrieve the credential/token as proof; no further cloud abuse unless RoE + authorized account.
- **IAM privesc:** enumerate the path; demonstrate the assumable role minimally; no destructive actions.
- **Storage:** list a benign object / show writability with a marker; remove it. No bulk download.
- **Feature gating/billing:** invoke one gated endpoint / show the bypass; no sustained abuse.
- **Subdomain takeover:** identify dangling record + reclaimable service; if reclaiming, benign marker only, no impersonation.
- **Resource/DoS:** generally out of scope on shared infra; if authorized, bounded burst only, never sustained.

---

*This methodology is bounded to authorized testing and honors the cloud shared-responsibility model and each provider's pentest policy. Tenant isolation is the marquee concern — prove it with your own tenants; SSRF is the keystone to the cloud control plane — prove retrieval, not abuse. Attribute every finding to the right responsibility owner, scope all proof to your tenants/authorized accounts, and map findings to OWASP + CWE + MITRE ATT&CK (Cloud). Mark mapping and exploitability uncertainty explicitly.*
