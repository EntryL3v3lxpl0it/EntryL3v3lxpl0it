# Fuzzing Methodology

**Document type:** Repeatable fuzzing campaign playbook
**Scope of applicability:** Source-available or binary targets you own or are authorized to test (lab, CTF, customer-approved code/assets)
**Standards referenced:** CWE (memory-safety classes), CVSS v3.1/v4.0, NIST SP 800-115 (testing), MITRE ATT&CK where a finding maps to a technique
**Toolchain assumed:** AFL++ (`afl-fuzz`, `afl-clang-fast`, `afl-cmin`, `afl-tmin`, `afl-showmap`, `afl-cov`), Clang/LLVM (`-fsanitize=fuzzer`, sanitizers, `llvm-cov`, `llvm-symbolizer`), `gdb`/`gef`, QEMU (AFL++ `qemu_mode` for binary-only). honggfuzz, `rr`, Valgrind, Dr. Memory, WinDbg, and CASR are referenced where relevant; install if not present.

> **Core principles baked into this methodology**
> - **Harnesses mirror the real source-code path.** A harness that calls a parser differently than production does will find harness bugs, not target bugs. Any deliberately reduced harness is labeled `// SYNTHETIC` and its divergence from the real path is documented.
> - **Separate evidence from hypothesis.** A crash is evidence of a fault; exploitability/impact is a hypothesis until demonstrated.
> - **Every crash gets a root cause or a documented reason it couldn't be obtained.** No "fuzzer found something" without source/binary-level explanation.
> - **Patch validation proves the invariant:** the same input must fail (sanitizer abort / crash) on the pre-fix build and pass on the post-fix build, and that input enters a regression corpus.
> - **Coverage and corpus are first-class artifacts**, versioned and re-minimized whenever the harness or instrumentation changes (reachability changes → "best corpus" changes).

## 0. Phase 0 — Scope & Authorization

**Goal:** Confirm you may test the target and define what "in scope" means for the campaign.

1. Authorization: ownership or written approval for the code/binary/service under test. Store in `00-scope/`.
2. Target inventory: source tree or binary, version/commit hash, build system, dependencies.
3. Attack surface declaration: which entrypoints consume untrusted input (file parsers, decoders, protocol handlers, importers, IPC, CLI/file readers).
4. Constraints: targets that touch shared infrastructure, network services that could be DoS'd, or rate-sensitive systems → fuzz a local copy or snapshot, not production.
5. Success criteria: e.g., "achieve ≥X% edge coverage of the parser, triage all unique crashes, deliver patch-validated findings."

**Validation gate:** authorization stored; target commit pinned; entrypoint list written to `00-scope/entrypoints.md`.

## 2. Phase 1 — Target Selection & Attack-Surface Mapping

**Goal:** Pick the highest-value fuzz targets. The selection criterion is the **intersection: complexity × exposure × impact** — not "lowest coverage" alone.

### 2.1 Enumerate untrusted-input entrypoints
List functions that accept attacker-controlled data: request handlers, file/format decoders, importers, protocol parsers, deserializers, CLI/file readers, hex/Base64/IP/message decoders.

### 2.2 Build a call graph and score nodes
Annotate each function node with: cyclomatic complexity, reachability from an untrusted entrypoint, blast radius (downstream dependents), and any corroborating findings (SAST/SARIF, audit notes, mutation-testing survivors).

```bash
# Example with Trailmark (graph + complexity + path queries)
uv pip install trailmark
trailmark analyze --complexity 10 ./target-src
```
```python
from trailmark.query.api import QueryEngine
engine = QueryEngine.from_directory("./target-src", language="c")
engine.preanalysis()
engine.complexity_hotspots(10)
engine.paths_between("handle_request", "parse_ipv6")   # confirm reachability from entrypoint
```

### 2.3 Prioritize
Rank candidates that combine **high cyclomatic complexity + confirmed reachability from untrusted input + high blast radius + corroborating evidence**. A complex parser/decoder with a confirmed attacker path beats an isolated internal helper with low coverage but no input path.

### 2.4 Use mutation-testing survivors as a target signal (not gospel)
For each survivor, ask via the graph: reachable from an attacker entrypoint? Constrained by stronger upstream invariants? In dead/formatting-only code vs. a boundary/overflow/parser path?
- Unreachable or structurally constrained → likely **equivalent mutant**, deprioritize.
- Reachable + touches boundary/overflow/security-critical arithmetic → promote to a harness, property test, or targeted edge-case vector.

**Validation gate:** ranked target list in `01-target-analysis/targets.md`, each with rationale (complexity, reachability path, impact).

## 3. Phase 2 — Harness Design

**Goal:** Drive the selected entrypoint **the way production does**, so coverage and crashes reflect real reachable behavior.

### 3.1 Engine selection
| Engine | Use when |
|--------|----------|
| **libFuzzer** (`clang -fsanitize=fuzzer`) | Source available, fast in-process, one entrypoint, easy sanitizer combo |
| **AFL++** (`afl-clang-fast`, persistent mode) | Source available; want CMPLOG/Redqueen, custom mutators, parallel; or binary-only via `qemu_mode` |
| **honggfuzz** | Alternative coverage engine; good hardware-assisted feedback (install separately) |
| **Snapshot/full-system** (Phase 6) | Deep state behind expensive setup; binary-only firmware/services |

### 3.2 libFuzzer harness (in-process)
```c
// 02-harness/fuzz_parse.c
#include <stdint.h>
#include <stddef.h>
#include "target.h"   // real header — call the SAME entry the app calls

int LLVMFuzzerTestOneInput(const uint8_t *data, size_t size) {
    // Mirror production: same init, same parse function, same teardown.
    target_ctx_t ctx;
    target_init(&ctx);
    target_parse(&ctx, data, size);   // the real reachable path
    target_free(&ctx);
    return 0;
}
```

### 3.3 AFL++ persistent-mode harness
```c
// 02-harness/fuzz_afl.c
#include <stdint.h>
#include <unistd.h>
__AFL_FUZZ_INIT();
int main(void) {
#ifdef __AFL_HAVE_MANUAL_CONTROL
    __AFL_INIT();
#endif
    unsigned char *buf = __AFL_FUZZ_TESTCASE_BUF;
    while (__AFL_LOOP(10000)) {
        int len = __AFL_FUZZ_TESTCASE_LEN;
        target_ctx_t ctx; target_init(&ctx);
        target_parse(&ctx, buf, len);
        target_free(&ctx);
    }
    return 0;
}
```

### 3.4 Anti-patterns to avoid
- **Shallow harness:** entering the target too late/early, or one oversized "do-everything" harness. Symptom: statically reachable functions never covered dynamically (see 3.6).
- **Harness-only parsing:** inventing a wire format the real code doesn't use hides logic bugs behind harness code. If the API consumes typed objects, fuzz the bytes and let the *real* deserializer run (or use struct-aware fuzzing where the framework supports it).
- **Unmarked synthetic harness:** any reduction from the real path must be labeled `// SYNTHETIC` and justified.

### 3.5 Stateful / typed APIs
- Multi-step protocols/parsers: treat the **message sequence** as the seed; keep valid prefixes stable; mutate the transition-driving message; cache server-generated IDs/nonces for dependent steps (see Phase 6.2).
- Typed Go APIs: struct-aware fuzzing (gosentry) when the code consumes structs/slices rather than scalars (Phase 6.5).

### 3.6 Harness introspection (find shallow harnesses early)
Before a long campaign, check for statically-reachable-but-dynamically-uncovered functions. Those indicate: harness enters too late/early, corpus missing a feature family, or a second harness is needed.
```bash
# OSS-Fuzz / Fuzz Introspector style triage
python3 infra/helper.py introspector <project> --seconds=30
python3 infra/helper.py introspector <project> --public-corpora
```

**Validation gate:** harness compiles, calls the real entrypoint, and a smoke run reaches the target function (confirm with a coverage probe in Phase 4.4). Synthetic deviations documented.

## 4. Phase 3 — Instrumentation & Sanitizer Build Matrix

**Goal:** Build the binaries the campaign needs: a fast fuzzing build, a CMPLOG build, sanitizer builds for bug surfacing, and a coverage build for measurement.

> Sanitizers are not mutually free to combine. ASan+UBSan combine well. **MSan is incompatible with ASan** and requires all dependencies instrumented — use a dedicated MSan build. LSan rides with ASan by default.

### 4.1 libFuzzer + sanitizers
```bash
# Primary surfacing build: ASan + UBSan
clang -g -O1 -fsanitize=fuzzer,address,undefined \
  -fno-omit-frame-pointer 02-harness/fuzz_parse.c target.c \
  -o 03-build/asan/fuzz_asan

# Dedicated MSan build (uninitialized reads) — separate, deps must be instrumented
clang -g -O1 -fsanitize=fuzzer,memory -fsanitize-memory-track-origins \
  02-harness/fuzz_parse.c target.c -o 03-build/msan/fuzz_msan
```

### 4.2 AFL++ build + CMPLOG (comparison-aware)
Hard comparisons (magic bytes, length/enum/checksum guards, `memcmp`/switch dispatch) cause plateaus. CMPLOG/Redqueen observes failed-comparison operands and biases mutations to satisfy them.
```bash
# Standard instrumented target
afl-clang-fast -g -O1 -o 03-build/afl/target 02-harness/fuzz_afl.c target.c

# CMPLOG variant
AFL_LLVM_CMPLOG=1 afl-clang-fast -g -O1 -o 03-build/cmplog/target.cmplog 02-harness/fuzz_afl.c target.c

# Autotools projects:
#   ./configure --cc=afl-clang-fast && make           -> normal
#   make clean && AFL_LLVM_CMPLOG=1 ./configure --cc=afl-clang-fast && make  -> cmplog
```

### 4.3 ASan runtime options (deterministic, symbolized)
```bash
export ASAN_OPTIONS=abort_on_error=1:symbolize=1:detect_leaks=1:halt_on_error=1
export UBSAN_OPTIONS=print_stacktrace=1:halt_on_error=1
export ASAN_SYMBOLIZER_PATH=$(command -v llvm-symbolizer)
```

### 4.4 Coverage build (measurement)
```bash
clang -g -O1 -fprofile-instr-generate -fcoverage-mapping \
  02-harness/fuzz_parse.c target.c -o 03-build/coverage/fuzz_cov
```

### 4.5 Binary-only targets
No source → AFL++ `qemu_mode` (`afl-fuzz -Q`), Frida mode, or move to snapshot fuzzing (Phase 6.4). Note the reduced feedback fidelity in the report.

**Validation gate:** all required build flavors present in `03-build/`, ASan build aborts on a known-bad input, coverage build produces a profile.

## 5. Phase 4 — Corpus & Seed Strategy

**Goal:** Provide structurally diverse, small, valid-prefix seeds and a dictionary so mutations land on meaningful fields.

### 5.1 Seed sourcing
Start from **real-world samples**: public corpora, the target ecosystem's file/test sets, captured traffic, crawled inputs. Real seeds carry the structure random bytes can't reach.

### 5.2 Corpus minimization (diversity over volume)
Prefer a **small set of structurally different seeds maximizing coverage** over a large pile of near-duplicates. Coverage-guided mutation is greedy and produces near-duplicate corpora over time; distill instead of hoarding.
```bash
# Distill to a minimal set preserving edge coverage
afl-cmin -i 04-corpus/seeds -o 04-corpus/minimized -- 03-build/afl/target @@
# Per-file minimization (shrink each input; keep mutations on meaningful fields)
for f in 04-corpus/minimized/*; do
  afl-tmin -i "$f" -o "$f.min" -- 03-build/afl/target @@
done
```
Keep seeds small enough that mutations hit fields, not filler. For grammar-aware targets, stop minimization at a **minimum token threshold** so enough surrounding structure remains mutation-friendly.

### 5.3 Dictionaries (often more valuable than a giant wordlist)
Extract grammar tokens, chunk names, protocol verbs, type tags, magic values, and delimiters from real samples, specs, or debug logs.
```bash
afl-fuzz -x 04-corpus/dict/target.dict ...   # AFL++
# libFuzzer: ./fuzz_asan -dict=04-corpus/dict/target.dict corpus/
```

### 5.4 Re-minimize on change
Re-run corpus minimization after any major harness/instrumentation change — the "best" corpus shifts when reachability changes.

**Validation gate:** `04-corpus/minimized/` distilled; dictionary present; smoke run shows seeds reach the target function in the coverage build.

## 6. Phase 5 — Running the Campaign

**Goal:** Execute coverage-guided fuzzing with comparison-aware mutation, parallelism, and live monitoring.

### 6.1 AFL++ with CMPLOG, parallel
```bash
# Main (deterministic) + CMPLOG attached via -c
afl-fuzz -i 04-corpus/minimized -o 05-campaign/out -M main \
  -x 04-corpus/dict/target.dict -c 03-build/cmplog/target.cmplog \
  -- 03-build/afl/target @@

# Secondaries (different schedules/mutators) in additional terminals
afl-fuzz -i 04-corpus/minimized -o 05-campaign/out -S sec1 -p explore -- 03-build/afl/target @@
afl-fuzz -i 04-corpus/minimized -o 05-campaign/out -S sec2 -p exploit -- 03-build/afl/target @@
```

### 6.2 libFuzzer
```bash
./03-build/asan/fuzz_asan -dict=04-corpus/dict/target.dict \
  -max_len=4096 -print_final_stats=1 \
  -artifact_prefix=06-crashes/raw/ 04-corpus/minimized/
```

### 6.3 Live monitoring & plateau response
```bash
afl-whatsup -s 05-campaign/out          # campaign health across instances
```
Decision points when coverage plateaus:
- Stuck on **hard comparisons** → confirm CMPLOG attached; enrich dictionary; solve earliest magic values first, then re-minimize so later stages start from valid prefixes.
- **Whole feature family uncovered** → corpus gap; add seeds for that feature, or split the harness (3.6).
- **Byte mutations dying in early syntax checks** → switch to grammar-based generation (Phase 6.3).

### 6.4 Coverage measurement
```bash
# Replay corpus through the coverage build, render a report
LLVM_PROFILE_FILE=05-campaign/coverage/cov.profraw 03-build/coverage/fuzz_cov 04-corpus/minimized/*
llvm-profdata merge -sparse 05-campaign/coverage/cov.profraw -o 05-campaign/coverage/cov.profdata
llvm-cov report 03-build/coverage/fuzz_cov -instr-profile=05-campaign/coverage/cov.profdata
# Or afl-cov for AFL++ output dirs.
```

**Validation gate:** campaign running with coverage feedback; coverage trend recorded; plateau strategy applied if growth stalls.

## 7. Phase 6 — Advanced Strategy Selection

**Goal:** Match the technique to why the naive campaign is failing. Decision matrix:

| Symptom / target property | Strategy |
|---------------------------|----------|
| Plateau on magic bytes / length / enum / checksum guards | **Comparison-aware** (AFL++ CMPLOG / Redqueen) + dictionary (6.1) |
| Multi-stage protocol / authenticated / "only-after-handshake" bug | **Stateful fuzzing** — sequence as seed, response codes as state oracle (6.2) |
| Byte mutations die in early syntax checks | **Grammar-based** generation (Nautilus / generative phase) (6.3) |
| Interesting code reachable only after expensive setup (boot/login/handshake) | **Snapshot fuzzing** — snapshot post-`recv`, mutate buffer in place (6.4) |
| Two implementations/parsers should agree | **Differential fuzzing** — compare outputs, not just crashes (6.6) |
| Go target with typed inputs / silent failures | **gosentry** — typed inputs, panic-on, race/leak replay (6.5) |
| Want generative novelty + coverage reuse on one box | **Jackalope-style** short-lived worker restarts (6.7) |

### 6.1 Comparison-aware (see 4.2/6.1) — solve earliest comparisons first, then re-minimize.

### 6.2 Stateful fuzzing
Keep valid prefix messages stable; focus mutation on the transition-driving message; cache identifiers/server values from prior responses; prefer per-message mutation/splicing over mutating the whole transcript as an opaque blob; use response codes/parser phases/returned object types as a cheap **state oracle** to prioritize sequences that progress deeper. (Tools: AFLNet for network protocols.)

### 6.3 Grammar-based fuzzing
Use when byte-level mutations mostly die in syntax checks. Keep the grammar focused on the **security-relevant subset**, not the full spec. Use large boundary values in terminals to stress integer/length/state-machine edges. The target still receives bytes/strings, so semantic checks remain inside the harnessed code. For dependency-heavy grammars (chained `document()`→`generate-id()`-style dataflow), hybridize generative and mutational phases or bias generation toward function-chaining patterns — pure coverage feedback won't preserve 3+-step dataflow chains.

### 6.4 Snapshot fuzzing
Run to the interesting state → snapshot memory+registers → write each mutated input directly into the guest/process input buffer → execute to crash/timeout → restore dirty pages → repeat. Practical trick: break right after `recv`/`read`/deserialization, note the input buffer address, snapshot there, mutate that buffer each iteration. Avoids re-paying setup cost; ideal for network services, firmware, post-auth surfaces, and binary-only targets hard to refactor into an in-process harness.

### 6.5 Go: gosentry (LibAFL backend for `testing.F`)
```bash
./bin/go test -fuzz=FuzzHarness --focus-on-new-code=false --catch-races=true --catch-leaks=true
```
Promote silent failures to findings: `--panic-on=pkg.Func` (treat `log.Fatal`-style paths as crashes), `--catch-races` (race-detector replay), `--catch-leaks` (goleak replay), LibAFL hang handling (keep infinite loops as findings), built-in arithmetic-overflow checks. Struct-aware fuzzing for typed APIs; Nautilus grammar mode (`--use-grammar --grammar=...`) for parser-heavy formats.

### 6.6 Differential fuzzing
Generate valid structured inputs, feed two parsers/clients/state engines, treat divergence as a finding:
```go
gotA, errA := ParseA(data)
gotB, errB := ParseB(data)
if (errA == nil) != (errB == nil) { t.Fatalf("parser disagreement: A=%v B=%v", errA, errB) }
```
Findings: one panics while the other rejects cleanly; accept/reject mismatch; different parse trees/decoded objects; divergent state transitions/nonces/balances/roots. Surfaces consensus mismatches, parser ambiguity, and spec-vs-implementation drift that crash-only fuzzing misses.

### 6.7 Single-machine diversity (Jackalope-style)
Restart short-lived workers against a persistent server: each worker starts from an empty corpus (`-in empty`), syncs after T seconds, runs T more on the combined corpus, syncs, exits. Fresh structures each generation while reusing accumulated coverage. Tune T to switch after a worker has found most "easy" coverage.

**Validation gate:** if the naive campaign plateaued, the matching advanced strategy is applied and coverage/finding growth re-measured.

## 8. Phase 7 — Crash Triage, Deduplication & Minimization

**Goal:** Convert raw crash files into a small set of unique, minimized, classified reproducers.

### 8.1 Reproduce under a sanitizer build
```bash
ASAN_OPTIONS=abort_on_error=1:symbolize=1 ./03-build/asan/fuzz_asan 06-crashes/raw/<crash>
```
Capture the full sanitizer report (error class, faulting access, stack) into `07-triage/`.

### 8.2 Deduplicate
Group by sanitizer error type + top-N stack frames (stack hash). AFL++ helpers and CASR automate this:
```bash
# AFL++ corpus/crash distillation
afl-cmin -C -i 06-crashes/raw -o 06-crashes/unique -- 03-build/afl/target @@
# CASR (optional): cluster + severity
casr-afl -i 05-campaign/out -o 07-triage/casr
casr-cluster -c 07-triage/casr 07-triage/casr-unique
```

### 8.3 Minimize each unique reproducer
```bash
afl-tmin -i 06-crashes/unique/<id> -o 06-crashes/minimized/<id>.min -- 03-build/afl/target @@
# libFuzzer: ./fuzz_asan -minimize_crash=1 -runs=100000 <crash>
```

### 8.4 Classify
Map each unique crash to a memory-safety class and CWE (state uncertainty rather than guessing):
| Sanitizer signal | Likely class | CWE |
|------------------|--------------|-----|
| `heap-buffer-overflow` (READ) | OOB read | CWE-125 |
| `heap-buffer-overflow` (WRITE) | OOB write | CWE-787 |
| `heap-use-after-free` | UAF | CWE-416 |
| `stack-buffer-overflow` | Stack OOB | CWE-121/787 |
| `attempting double-free` | Double free | CWE-415 |
| UBSan signed-overflow / shift | Integer error | CWE-190/191 |
| MSan use-of-uninitialized | Uninit read | CWE-457 |
| `SEGV` on unknown address | Null/wild deref | CWE-476 (if null) |
| Hang / timeout | DoS | CWE-834/400 |

**Validation gate:** `06-crashes/minimized/` holds one minimized reproducer per unique fault, each with a sanitizer report and provisional CWE in `07-triage/`.

## 9. Phase 8 — Root-Cause Analysis

**Goal:** Explain *why* each unique crash happens at the source/binary level — the difference between "fuzzer found a crash" and an actionable finding.

1. Reproduce under a debugger; correlate the faulting instruction/line to source.
```bash
gdb -q --args 03-build/asan/fuzz_asan 06-crashes/minimized/<id>.min
# gef: pattern, registers, backtrace, dereference faulting pointer
```
2. For non-deterministic or hard-to-catch bugs, record/replay:
```bash
rr record 03-build/asan/fuzz_asan 06-crashes/minimized/<id>.min   # then: rr replay (reverse-continue)
```
   Windows targets: WinDbg + Time Travel Debugging. Heap/uninit edge cases: Valgrind (memcheck) or Dr. Memory for a second opinion.
3. Trace the tainted field from the input buffer to the faulting operation. Mirror the *Bug Hunter's Diary* approach: identify the exact byte/field that drives the fault (e.g., a size/length field whose attacker-controlled value reaches a copy), and confirm by editing that field to flip the behavior.
4. Identify the root cause (missing bounds check, signed/unsigned conversion, lifetime error, integer overflow feeding an allocation/copy) and the **preconditions** for reachability from the real entrypoint.
5. Assess exploitability separately and evidence-based: attacker-controlled offset/length/pointer, allocation primitive, write vs. read, mitigations present. Mark **CONFIRMED vs. INFERRED**.

**Validation gate:** each unique crash has a written root cause, the controlling input field identified, reachability from a real entrypoint confirmed or noted as harness-only, and an evidence-based exploitability statement.

## 10. Phase 9 — Patch Validation

**Goal:** Prove the fix addresses the invariant, not just the one input.

1. Build pre-fix and post-fix targets with the same sanitizer flags.
```bash
# before/ = vulnerable commit, after/ = patched commit
clang -g -O1 -fsanitize=address,undefined 02-harness/fuzz_parse.c before/target.c -o 08-patch-validation/before
clang -g -O1 -fsanitize=address,undefined 02-harness/fuzz_parse.c after/target.c  -o 08-patch-validation/after
```
2. Run the minimized reproducer against both. The invariant: **fails before, passes after.**
```bash
ASAN_OPTIONS=abort_on_error=1 ./08-patch-validation/before 06-crashes/minimized/<id>.min ; echo "before rc=$?"  # expect abort
ASAN_OPTIONS=abort_on_error=1 ./08-patch-validation/after  06-crashes/minimized/<id>.min ; echo "after rc=$?"   # expect clean
```
3. Generalize beyond the single input: add the reproducer to a **regression corpus**, then re-fuzz the patched build briefly to confirm the fix doesn't merely move the crash one byte over and that no adjacent variant reproduces.
```bash
cp 06-crashes/minimized/<id>.min 04-corpus/regression/
afl-fuzz -i 04-corpus/regression -o 08-patch-validation/refuzz -V 600 -- after/target @@   # timeboxed
```
4. If a property/invariant test is cheaper than a full reproducer (e.g., the function must reject length > buffer), add that test too.

**Validation gate:** documented before-fails / after-passes result per finding, reproducer in the regression corpus, short re-fuzz of the patched build clean.

## 11. Phase 10 — Reporting

**Goal:** Deliver evidence-based findings traceable to reproducers and root cause, readable by a PM and defensible to engineers.

### 11.1 Finding template
```markdown
### [Finding ID] <Title>

**Summary:** <Plain-language: what breaks, why it matters.>

**Affected component:** <module/function, version/commit>
**Attack surface / entry point:** <real entrypoint that reaches the fault>
**Weakness classification:** <memory-safety / logic / DoS class>
**CWE mapping:** CWE-### <name>   (mark "uncertain" if not confidently mappable)
**CVSS:** <vector> → <score/severity>   (omit/mark "not scored" if unsupportable)

**Preconditions:** <Input shape, reachable config, position needed.>

**Evidence:**
- Reproducer: 06-crashes/minimized/<id>.min  (sha256: ...)
- Sanitizer report: <error class, faulting access, top frames>
- Coverage/debugger artifacts: <paths>

**Root cause:** <Source/binary-level explanation; the controlling input field.>

**Impact:** <Evidence-bounded consequence: read vs. write, control over offset/length/pointer, DoS, info leak. Separate CONFIRMED from INFERRED.>

**Exploitability:** <Reliability, attacker control, mitigations present. CONFIRMED vs. INFERRED.>

**Reproduction steps:**
1. Build: <command>
2. Run: ASAN_OPTIONS=... ./fuzz_asan <reproducer>
3. Observe: <sanitizer abort + stack>

**Expected vs. observed:**
- Expected: <safe handling — reject/clamp/error>
- Observed: <sanitizer-detected fault>

**Fix recommendation:** <Specific: bounds check, type/width fix, lifetime correction, integer guard.>

**Validation after fix:** <Reproducer fails on pre-fix build, passes on post-fix; regression corpus entry; invariant test.>

**Residual risk:** <Adjacent surface, related paths not yet covered, coverage gaps.>
```

### 11.2 Campaign summary section
Report coverage achieved (with the measurement method), corpus/dictionary provenance, engines/strategies used, total executions, unique crashes, and **explicit coverage gaps** (untested feature families / functions) so readers know what was *not* exercised.

**Validation gate:** every finding links to a hashed reproducer + sanitizer evidence + root cause + patch-validation result; coverage gaps stated.


## 15. Appendix C — Per-Target Checklist
```
[ ] Entrypoint selected via complexity × exposure × impact
[ ] Harness mirrors real path (synthetic deviations labeled)
[ ] Harness introspection: no large statically-reachable-but-uncovered region
[ ] Builds: afl + cmplog + asan/ubsan (+ msan if uninit suspected) + coverage
[ ] Seeds from real samples; corpus distilled (afl-cmin); per-file min (afl-tmin)
[ ] Dictionary from spec/samples/logs
[ ] Campaign running with CMPLOG + parallel instances
[ ] Coverage measured + trend recorded; plateau → matched advanced strategy
[ ] Crashes: reproduced under ASan, deduped, minimized, CWE-classified
[ ] Root cause + controlling field + reachability confirmed
[ ] Patch validation: before-fails / after-passes + regression corpus
[ ] Finding written with hashed reproducer + sanitizer evidence
[ ] Coverage gaps documented
```

---

*This methodology is bounded to owned or authorized targets. Coverage numbers and crash counts are campaign artifacts, not assurance guarantees — always report what was not exercised. Promote nothing to a finding without a reproducer, a root cause, and a patch-validated invariant; mark CWE/CVSS and exploitability uncertainty explicitly.*
