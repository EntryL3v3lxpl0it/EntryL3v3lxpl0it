# Penetration Testing Methodology Suite — Master Index

**Purpose:** The single entry point to the methodology suite. Use it to (1) pick the right playbook(s) for an engagement, (2) follow the handoffs between domains, and (3) apply the shared conventions every playbook assumes. The individual playbooks hold the depth; this index holds the map.

**Suite version:** 1.0 · **Documents:** 10 · **Toolchain baseline:** Kali Linux build per the maintained tool inventory.

## 2. Engagement Selector

Given the scope, open these playbooks in this order. The **Lifecycle (V)** always runs per finding regardless of domain.

| Engagement scope | Open, in order |
|------------------|----------------|
| Internet-facing network/perimeter | 1 → (foothold) 2 → 3 → V |
| Internal network / "assume breach" | 2 → 3 → (hybrid) 6 → V |
| Full domain / forest security review | 3 (with 2 for the lifecycle) → (hybrid) 6 → V |
| Web app (server-rendered / API-backed) | 4 → 5 → (white-box) source-review in 4/5 → (parser) 9 → V |
| API / microservices platform | 5 → (SSRF→internal) 2/6 → (containers/K8s) 5§7 → V |
| Multi-tenant SaaS product | 6 → 4 + 5 (app/API surface) → (SSRF→cloud) 6§6 → V |
| Mobile app (iOS/Android) | 8 → (backend) 5 + 4 → (native libs) 9 → V |
| Phishing / human-layer resilience | 8 → (initial access) 2 → 3 → V |
| Red team (full kill chain) | 8 → 1/2 → 3 → 6 → V throughout |
| Binary / parser / protocol robustness | 9 → V (per crash) |
| Single vulnerability, end-to-end | V (pulls domain playbook for the relevant surface) |

**Rule of thumb:** domain playbooks find and prove *reachability*; the Lifecycle takes each confirmed item from validation → safe PoC → impact → remediation → regression. Never close a finding outside the Lifecycle.

## 3. The Lifecycle Vertical (applies to every finding)

Every confirmed candidate from any domain playbook enters the same state machine (playbook V):

```
CANDIDATE → VALIDATED → ROOT-CAUSED → EXPLOITABILITY-ASSESSED →
PoC-PROVEN → IMPACT-DEMONSTRATED → REMEDIATION-RECOMMENDED →
FIX-VALIDATED → REGRESSION-PROTECTED → CLOSED
```
Off-ramps: **FALSE-POSITIVE**, **NOT-EXPLOITABLE/INFORMATIONAL**, **ACCEPTED-RISK** (each with documented rationale).

The domain playbooks produce a **candidate + evidence**; V enforces "a candidate is not a finding," builds the **safe PoC**, scores **business impact**, and guarantees the fix **fails-before / passes-after** with a **regression artifact**. The finding template in §6 is shared by all playbooks and is completed at V's Impact phase.

**Key handoffs in words:**
- **External → Internal:** a perimeter foothold/pivot becomes the assumed-breach starting posture.
- **Internal ↔ AD-Enterprise:** (2) is the *engagement lifecycle*; (3) is the *AD attack-surface taxonomy/depth*. Run (2) for flow, (3) for Kerberos/ACL/ADCS/GPO/trust/hybrid depth.
- **AD-Enterprise → SaaS/Cloud:** Entra Connect / federation is the on-prem↔cloud bridge into (6).
- **Web → API:** the app's API surface (and OpenAPI/GraphQL) drives (5); UI controls are re-tested at the API layer.
- **API/SaaS SSRF → Internal/Cloud:** SSRF is the lateral primitive — into internal services (2) or the cloud control plane (6 §6, IMDS→IAM).
- **Mobile → API/Web:** the app is a thick client; intercepted traffic is the backend attack surface for (5)/(4).
- **Any parser/native/protocol surface → Fuzzing (9):** custom decoders/`.so`/Mach-O/message parsers move to a fuzzing campaign.
- **SE → Internal → AD:** social engineering is the initial-access leg feeding (2)→(3) in a red team.
- **Everything → Lifecycle (V):** universal vertical; no finding closes elsewhere.

## 5. Shared Conventions (stated once; assumed by all playbooks)

### 5.1 Engagement folder scheme
Every engagement scaffolds a numbered, phase-aligned tree:
```
<PREFIX>-YYYYMMDD-CLIENT/
  00-scope/                 # authorization, RoE, exclusions, identities
  0N-<phase>/ ...           # one dir per methodology phase
  07-evidence/{screenshots,raw,...}   # immutable record
  10-cleanup/ or 10-data-handling/    # where state is created (2,3,6,8)
  99-loot/ or 99-secure-store/        # retrieved data, per data rules
```
`<PREFIX>` per the table in §1. Engagements that **create state** (Internal, AD-Enterprise, SaaS, SE) carry a mandatory `change-log` and cleanup phase.

### 5.2 Evidence naming & integrity
- **Naming:** `<phase>_<target-or-asset>_<tool>_<UTC>.<ext>` (e.g., `03_198.51.100.10_smbmap_20250625T1432Z.txt`).
- **Integrity:** hash raw evidence at collection (`sha256sum`); raw files are never edited; the finding cites artifacts by filename + hash.
- **Separation:** keep **evidence (observed)** distinct from **hypothesis (inferred)** in every record.

### 5.3 Phase structure
Each phase ends with a **validation gate** (explicit exit criteria) and contains **decision points** (forks, e.g., "RoE prohibits X → cap at Y"). Forward momentum is gated, not continuous.

### 5.4 Safety constructs (common vocabulary)
| Construct | Meaning | Where strongest |
|-----------|---------|-----------------|
| **Authorization hard-blocker** | Phase 0 will not pass without signed scope/RoE | all; expanded in SE (exec+HR+Legal) |
| **Safe PoC** | minimum action to prove the primitive; benign markers, OOB, reversible | all; canonical in V |
| **Evidence vs hypothesis / CONFIRMED vs INFERRED** | never assert impact you didn't observe | all |
| **Change-log + cleanup** | log and revert every created object | 2, 3, 6, 8 |
| **Gating** | destructive / persistence / dominance only with explicit RoE | 2, 3 (golden ticket, DCSync-persist), 6 |
| **Lockout awareness** | model policy before any spraying | 1, 2, 3, 4, 5 |
| **Tenancy / Tier-0 / shared-responsibility scoring** | score paths by the boundary crossed | 3 (Tier 0), 6 (tenancy + responsibility owner) |
| **Duty of care** | protect the people tested; aggregate/anonymize | 8 |

### 5.5 Standards applied
- **Universal:** CWE, CVSS v3.1/v4.0, MITRE ATT&CK, NIST SP 800-115.
- **Domain overlays:** OWASP WSTG/Top 10 (4), OWASP API Top 10 (5), OWASP MASVS/MASTG (8-mobile), Microsoft Tier 0 model (3), cloud shared-responsibility + provider pentest policy (6), sanitizer/CWE memory-safety classes (9), SE ethical/legal frameworks (8-SE).
- **Mark uncertainty** rather than guessing a CWE/CVSS/ATT&CK mapping.

## 6. Canonical Finding Template

All playbooks complete this at the Lifecycle Impact phase. Domain-specific lines (WSTG ID / OWASP API / MASVS / tiering impact / tenancy impact / shared-responsibility owner / detection expectation) are **added** by the relevant playbook; the core is constant.

```markdown
### [Finding ID] <Title>
**Summary:** <plain-language risk for a PM>
**Affected component:** <...>     **Attack surface / entry point:** <...>
**Weakness classification:** <...>
**CWE:** CWE-### (uncertain?)     **CVSS:** <vector> → <severity> (or "not scored" + why)
**MITRE ATT&CK:** <Txxxx(.xxx)>   [+ domain overlay: WSTG/API##/MASVS/tiering/tenancy/owner/detection]
**Preconditions:** <auth/role/position/config/timing>
**Evidence:** <observed only; artifacts by filename + sha256; source file:line if white-box>
**Root cause:** <the defect + controlling input/config>
**Impact:** <evidence-bounded; CONFIRMED vs INFERRED; chains/blast radius>
**Exploitability:** <reliability, preconditions, mitigations; CONFIRMED vs INFERRED>
**Reproduction steps:** <exact, repeatable>
**Expected vs. observed:** <secure behavior vs what happened>
**Fix recommendation:** <root-cause fix + responsibility owner + defense-in-depth>
**Validation after fix:** <invariant: fails before / passes after>
**Residual risk:** <what remains; compensating controls>
```

## 7. How to Use This Suite

1. **Scope the engagement** → pick playbook(s) from §2; scaffold the folder tree (§5.1).
2. **Run the domain playbook(s)** phase by phase, honoring validation gates and decision points; follow §4 handoffs when a surface crosses domains.
3. **Route every confirmed candidate to the Lifecycle (V)** — validate, safe-PoC, score impact (canonical template §6), recommend the fix, prove fix-before/after, leave a regression artifact.
4. **Clean up** (change-log revert / data destruction) where the engagement created state.
5. **Report** using the per-playbook structure; cross-reference chains that span domains.
