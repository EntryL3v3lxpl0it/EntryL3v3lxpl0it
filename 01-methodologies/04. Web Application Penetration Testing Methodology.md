# Web Application Penetration Testing Methodology

**Document type:** Repeatable assessment playbook (black-box + white-box / source-assisted)
**Scope of applicability:** Web applications, APIs, and their supporting surfaces you are authorized to test — Internet-facing or internal, with or without source access
**Standards referenced:** OWASP Web Security Testing Guide (WSTG v4.2), OWASP Top 10 (2021) / API Security Top 10 (2023), OWASP ASVS, CWE, CVSS v3.1/v4.0, NIST SP 800-115
**Toolchain assumed:** This Kali build — Burp Suite / ZAP / Caido (intercepting proxies), `ffuf`/`feroxbuster`/`gobuster`/`dirsearch` (discovery), `sqlmap`/`commix`/`tinja`/`sstimap`/`xsstrike` (injection), `nuclei`/`nikto`/`wapiti`/`wpscan` (scanners), `whatweb`/`wafw00f`/`testssl.sh` (fingerprint/TLS), `arjun` (param discovery), `hakrawler`/`gospider`/`getallurls`/`waybackpy` (crawl), `phpggc` (deser gadgets), `graudit` (source grep), `seclists`/`payloadsallthethings` (payloads), `mitmproxy`/`websocat`/`wscat` (proxy/WS). `semgrep`, `ysoserial`, `jwt_tool`, `interactsh`/Burp Collaborator referenced where relevant (install / Burp Pro).

> **Operating constraints baked into this methodology**
> - **Two tracks, one model.** Black-box dynamic testing and white-box source-assisted review run in parallel and cross-feed: source review finds sinks → dynamic testing proves reachability/impact; dynamic anomalies → targeted source lookups for root cause.
> - **Map every request to evidence.** Findings ship with the exact request/response (and source location, when available). Separate **evidence** (observed) from **hypothesis** (inferred).
> - **Non-destructive by default.** Prove impact with the minimum action: benign markers, read-only proof, out-of-band (OOB) callbacks rather than data destruction. No bulk PII extraction — schema/row-count proof instead.
> - **Lockout & rate awareness.** Model lockout before auth testing; spray within budget; throttle automated tools against production.
> - **Authorization tests use multiple identities.** BOLA/BFLA/privilege-escalation require ≥2 accounts per role (or one-per-role matrix); never test access control with a single session.
> - Map findings to **WSTG ID + CWE + OWASP Top 10**; don't fabricate mappings — mark uncertainty.

## 0. Phase 0 — Pre-Engagement & Scoping

**Goal:** Fix authorization, scope, test data, and safety rails before the first request.

### 0.1 Authorization & scope (hard blocker) — stored in `00-scope/`
1. Signed authorization naming the in-scope apps, hostnames, API base URLs, and ownership attestation.
2. **Environment:** test/staging vs production. Prefer a non-prod copy for destructive or load-bearing tests; record version/commit if source-backed.
3. **Exclusions:** out-of-scope hosts/paths, third-party SSO/IdP and payment processors (need their authorization), fragile workflows.
4. **Credentials & roles:** provided accounts for **every role** (anon, user, privileged, admin) — minimum two per role for authorization testing. Test data that may be created/modified.
5. **Source access (white-box):** repository/commit, build/run instructions, framework(s), language(s), ORM, templating engine. Determines whether Track B (source review) runs.
6. **RoE intensity:** is automated scanning allowed against prod? Destructive payloads? Account lockout testing? Real vs marker data for exfil proof?
7. **WAF/CDN:** in path? Test through it (real-world) and/or with an allow-listed origin (coverage)? Source IPs to allow-list.
8. Emergency stop + contacts.

### 0.2 Decision point
- Production-only + no destructive RoE → cap injection/file-write tests at non-destructive PoC (read-only, OOB) and flag where deeper proof needs a staging copy.
- Source provided → enable Track B; schedule source review to lead dynamic testing per surface.

**Validation gate:** authorization stored, role accounts received, environment + source posture set, lockout budget defined, WAF strategy agreed.

## 2. Phase 1 — Recon & Information Gathering (WSTG-INFO)

**Goal:** Build the technology and exposure picture. MITRE ATT&CK: Reconnaissance (TA0043).

### 2.1 Surface & technology fingerprint
```bash
# Live web surfaces (incl. subdomains in scope)
httpx -l 01-recon/hosts.txt -title -status-code -tech-detect -web-server -o 01-recon/httpx.txt
whatweb -a3 https://target/ | tee 01-recon/whatweb.txt
wafw00f https://target/ | tee 01-recon/wafw00f.txt
testssl.sh --quiet --color 0 https://target/ > 01-recon/testssl.txt   # TLS/cipher/cert posture
```

### 2.2 Historical & client-side intel
```bash
getallurls target | tee 01-recon/gau.txt                  # known URLs from passive sources
waybackpy --url target --known_urls > 01-recon/wayback.txt
# Pull and review JavaScript for endpoints, secrets, params, routes
hakrawler -url https://target/ -js | tee 02-mapping/js/endpoints.txt
```
Review JS bundles for API routes, hidden parameters, feature flags, hard-coded secrets/keys, and client-side auth logic — these seed Phase 2/4/8.

### 2.3 (Track B) Source-side recon
If source provided: identify framework, router, ORM, templating engine, auth library, and input-handling middleware. Locate the route table (maps URLs → controllers → functions) — this is the master index for targeted source review.

**Validation gate:** tech stack, TLS posture, WAF presence, and an initial endpoint/JS inventory recorded; (Track B) route table located.

## 3. Phase 2 — Mapping: Content, Parameters & Functionality (WSTG-INFO/CONF)

**Goal:** Enumerate the full attack surface — every endpoint, parameter, and role-gated function. *Respect your methodology here; unmapped surface is untested surface.*

### 3.1 Crawl + forced browsing
```bash
# Authenticated crawl via Burp/ZAP (configure session) + automated discovery
feroxbuster -u https://target/ -w /usr/share/seclists/Discovery/Web-Content/raft-medium-directories.txt \
  -t 20 --rate-limit 50 -o 02-mapping/content/ferox.txt
ffuf -u https://target/FUZZ -w /usr/share/seclists/Discovery/Web-Content/raft-medium-files.txt \
  -mc 200,201,204,301,302,307,401,403 -o 02-mapping/content/ffuf.json
```

### 3.2 Parameter discovery
```bash
arjun -u https://target/endpoint -oT 02-mapping/params/arjun_endpoint.txt   # hidden params
# Burp param miner (Pro) for headers/params/cache keys; record findings.
```

### 3.3 API specification & endpoints
```bash
# If an OpenAPI/Swagger/GraphQL schema is exposed, capture and parse it
curl -s https://target/openapi.json -o 02-mapping/apis/openapi.json
# GraphQL introspection (if enabled): record full schema for Phase 8.
```

### 3.4 Build the role-access matrix
For every discovered function/endpoint, record which role(s) *should* access it. This matrix drives authorization testing (Phase 5) — you'll later attempt each cell from a lower-privilege identity.

**Validation gate:** endpoint inventory + parameter list + API spec captured; role-access matrix drafted in `02-mapping/`. Every subsequent test references this inventory so nothing is skipped.

## 4. Phase 3 — Configuration & Deployment Testing (WSTG-CONF, ERRH, CRYP)

**Goal:** Find weaknesses in how the app is deployed and configured.

Checks (capture each into `03-config/`):
1. **TLS** (from 2.1): protocol/cipher support, cert validity, HSTS.
2. **Security headers:** CSP, X-Content-Type-Options, X-Frame-Options/frame-ancestors, Referrer-Policy. Missing/weak → note.
3. **HTTP methods:** `OPTIONS`, `PUT`/`DELETE`/`TRACE` enabled? `curl -X OPTIONS -i https://target/`.
4. **Cookie attributes:** `Secure`, `HttpOnly`, `SameSite`, scope/path.
5. **CORS:** reflective `Access-Control-Allow-Origin` with credentials → test with a foreign `Origin`.
6. **Exposed files/dirs:** `.git/`, backups, `/actuator`, `.env`, source maps, admin consoles, directory listing.
```bash
nuclei -u https://target/ -t http/exposures -t http/misconfiguration -o 03-config/nuclei_misconfig.txt
nikto -h https://target/ -o 03-config/nikto.txt
```
7. **Error handling:** force errors (bad types, oversized input) → stack traces / framework versions / path disclosure.

**Validation gate:** config findings documented with the request that proves each; version/tech leaks catalogued.

## 5. Phase 4 — Identity, Authentication & Session Management (WSTG-IDNT/ATHN/SESS)

**Goal:** Test account lifecycle, authentication strength/bypass, and session handling. MITRE ATT&CK: T1078, T1110.

### 5.1 Identity & user enumeration
- Registration/login/reset responses or timing that reveal valid usernames (WSTG-IDNT-004).
- Account-provisioning and role-assignment flaws.

### 5.2 Authentication (lockout-aware)
```bash
# Username harvesting + spraying within lockout budget (model policy first)
ffuf -u https://target/login -X POST -d 'user=FUZZ&pass=Spring2025!' \
  -w users.txt -H 'Content-Type: application/x-www-form-urlencoded' \
  -mc all -ac -o 04-authn/spray.json
```
- **Auth bypass logic** (Track B is decisive here): SQLi in login (OSWE-style auth bypass), logic flaws, response/parameter tampering, default/weak credentials, OAuth/SAML/JWT flaws.
- **Password reset logic:** predictable tokens ("when random isn't"), token leakage via Host header, account-determination + timing, reset for another user.
- **MFA:** bypass, brute-forceable OTP, backup-code weakness.

### 5.3 JWT / tokens
```bash
# JWT analysis: alg confusion (none/HS-vs-RS), weak secret, claim tampering
jwt_tool <token> -M at -t https://target/api   # if installed; else manual + Burp
hashcat -m 16500 jwt.txt wordlist.txt          # crack weak HMAC secret
```

### 5.4 Session management
- Token randomness/entropy, predictability, fixation, lifetime, invalidation on logout/reset, concurrent sessions.
- **CSRF:** state-changing requests lacking anti-CSRF tokens or with bypassable validation (SameSite reliance, token not bound to session).

**Validation gate:** each auth/session finding has reproduction + observed-vs-expected; (Track B) root cause located in auth code where source available.

## 6. Phase 5 — Authorization (WSTG-ATHZ)

**Goal:** Prove access-control failures using **multiple identities**. OWASP A01. *This is the highest-value, most-missed category — drive it from the role-access matrix.*

### 6.1 BOLA / IDOR (object-level)
For each object reference (numeric ID, UUID, filename, GUID), replay a low-privilege user's request substituting another user's object identifier; confirm cross-tenant/cross-user access.
- Static-file IDOR and DB-object (ID-based) IDOR both apply.
- Automate the matrix with Burp (Autorize/Auth Analyzer) — request as user A, replay as user B, diff responses.

### 6.2 BFLA (function-level) & privilege escalation
- Access admin/privileged endpoints from a lower-privilege session (vertical).
- Force-browse unlinked functions from the inventory; verify server-side enforcement, not just UI hiding.

### 6.3 Path traversal & mass assignment
- Directory/path traversal to read config/source (OSWA case-study pattern: traversal → config file → admin access).
- Mass assignment / autobinding: add privileged fields (`role`, `isAdmin`) to update requests.

**Validation gate:** every access-control finding demonstrated with two distinct identities (or anon vs auth); response diff captured proving unauthorized access.

## 7. Phase 6 — Input Validation & Injection (WSTG-INPV)

**Goal:** Find and prove server-side and client-side injection. OWASP A03. **Non-destructive PoC; OOB over data damage.**

For each input (params, headers, cookies, JSON/XML fields, file uploads, path segments), test the relevant class. Track B accelerates this: locate the sink, then craft the input that reaches it.

### 7.1 SQL / NoSQL injection
```bash
# Confirm manually first (error/boolean/time-based), then automate carefully
sqlmap -r 07-evidence/raw/req.txt --batch --level 3 --risk 2 --technique=BEUST \
  --output-dir=06-injection/sqli   # avoid --dump on prod; prove with --current-user/--dbs
# NoSQL: operator injection ($ne, $gt), auth bypass via JSON body
```
- Proof: extract `current_user`/version/one row — not bulk data. SQLi→RCE (stacked queries, `xp_cmdshell`, file write) only on authorized non-prod, as bounded PoC.

### 7.2 Command injection
```bash
commix -r 07-evidence/raw/req.txt --output-dir=06-injection/cmdi   # confirm, then minimal proof
# OOB confirmation: payload triggers DNS/HTTP callback to Collaborator/interactsh
```
Prove with `id`/`hostname` or an OOB callback; stop there.

### 7.3 SSTI
```bash
sstimap -u 'https://target/page?name=*' -o 06-injection/ssti   # identify engine, confirm eval
# tinja for template-injection discovery; manual: {{7*7}}, ${7*7}, <%= 7*7 %> per engine
```
Discover the rendering function (Track B), confirm expression evaluation, then demonstrate minimal RCE primitive per engine (OSWE ERPNext-style).

### 7.4 XSS (stored / reflected / DOM)
- Reflected/stored: confirm with a benign marker payload that proves script execution context; capture screenshot.
- DOM: trace source→sink in JS (`location`/`postMessage`/`innerHTML`); Burp DOM Invader.
```bash
xsstrike -u 'https://target/search?q=test' --crawl   # candidate generation; validate manually
```
Use a benign PoC (`alert(document.domain)` / marker), not credential theft against real users.

### 7.5 SSRF
- Identify URL-consuming params (webhooks, importers, PDF/preview, image fetch).
- Confirm via OOB callback; escalate to loopback/internal/metadata (cloud) **read-only** proof.
- Test alternative URI schemes (`file://`, `gopher://`, `dict://`) and blind-SSRF port/subnet probing where authorized (OSWE Directus-style).

### 7.6 XXE
- XML entry points (SOAP, file upload, SAML, document import): test external-entity expansion → file read / SSRF; blind XXE via OOB DTD.

### 7.7 File inclusion / upload
- LFI/RFI: path traversal to source/config, log poisoning, PHP wrappers (`php://filter` source read).
- Upload: bypass type/extension checks; prove with a benign file, not a live web shell on prod.

### 7.8 Other input classes
Host-header injection, CRLF/response splitting (`crlfuzz`), HTTP request smuggling (authorized, careful), prototype pollution (client + server JS inheritance abuse), open redirect.

### 7.9 Insecure deserialization
- **PHP:** gadget chains with `phpggc` (e.g., `phpggc Framework/Chain` → payload) where a serialized blob is accepted.
- **Java:** `ysoserial` gadget chains (install) for vulnerable endpoints.
- **.NET:** `ObjectDataProvider`/`ExpandedWrapper` chains (OSWE-style) where a type-confused deserializer is reachable.
- **Python:** pickle RCE where untrusted input is unpickled.
Prove with an OOB callback or benign command; document the gadget and the reaching sink.

**Validation gate:** each injection finding confirmed (not scanner-only), with a non-destructive PoC; (Track B) the vulnerable sink and tainted source identified.

## 8. Phase 7 — Business Logic & Client-Side (WSTG-BUSL/CLNT)

**Goal:** Find flaws that aren't input-validation bugs — they require understanding the workflow.

### 8.1 Business logic
- Workflow/sequence bypass (skip payment, reuse one-time tokens, negative quantities, price/qty tampering).
- **Race conditions:** concurrent requests to double-spend, bypass limits, or exploit TOCTOU (Burp Turbo Intruder / parallel requests).
- Authorization-in-workflow: can step N be reached without completing step N-1?

### 8.2 Client-side
- DOM XSS (7.4), `postMessage` origin validation, clickjacking (frame-ability vs CSP `frame-ancestors`), sensitive data in client storage, client-side-only access control.

**Validation gate:** logic/client findings reproduced with the exact request sequence; impact tied to a business consequence.

---

## 9. Phase 8 — API & Modern Surfaces (OWASP API Top 10)

**Goal:** Test REST/GraphQL/WebSocket surfaces, which often expose the matrix more directly than the UI.

### 9.1 REST
- Drive every operation from the OpenAPI spec; test BOLA/BFLA per endpoint (Phase 5), mass assignment, excessive data exposure, missing rate limiting, and improper inventory (old API versions `/v1` vs `/v2`).

### 9.2 GraphQL
- Introspection enabled? Map the schema; test field-level authorization, query batching/aliasing for brute force, depth/complexity DoS, and injection through resolvers.

### 9.3 WebSocket
```bash
wscat -c wss://target/ws        # manual interaction; or websocat for scripting
```
- Auth on the WS handshake, origin validation, message-level authorization, injection via WS messages.

**Validation gate:** API/GraphQL/WS findings captured with raw frames/requests; authorization re-tested at the API layer (UI controls are not server controls).

## 10. Phase 9 — White-Box / Source-Assisted Review (Track B)

**Goal:** Systematic source review that finds what black-box misses and supplies root cause for every dynamic finding. Runs in parallel with Phases 2–9; this phase is the *method*.

### 10.1 Map trust boundaries: sources → sinks
1. **Sources** (attacker-controlled input): request params/body/headers/cookies, file uploads, message queues, imported docs, DB values reflected back.
2. **Sinks** (dangerous operations): SQL query builders, command exec, template render, deserializers, file ops, HTTP clients (SSRF), XML parsers, redirect/HTML output.
3. **Trace taint** from each source to each sink through controllers/services; an unsanitized path is a candidate finding.

### 10.2 Dangerous-function sweep (per language)
```bash
graudit -d <db> ./src                 # grep-based audit signatures (php/python/js/java/etc.)
semgrep --config auto ./src           # if installed; richer taint rules
```
Targets by ecosystem: raw string-concatenated SQL; `exec`/`system`/`popen`/`Runtime.exec`; `eval`/template render with user data; `unserialize`/`pickle.loads`/`readObject`/`BinaryFormatter`; XML parsers with external entities enabled; HTTP clients fed user URLs; redirects with user-controlled targets.

### 10.3 Auth & access-control review
- Where is authentication enforced, and is it enforced on **every** privileged route (middleware vs per-handler gaps)? OSWE-style auth-bypass bugs live here (e.g., SQLi in the login query, logic that trusts a client value).
- Object ownership checks at the data layer (the root cause of BOLA/IDOR).

### 10.4 Correlate with dynamic findings (SAST-on-graph discipline)
Project source-review hits and any SAST/SARIF output onto the route/call structure; for each, ask: reachable from an attacker-controlled entrypoint? Constrained by an upstream check? On a path to a security-critical sink? Promote reachable + unconstrained hits to Phase 10 PoCs; deprioritize unreachable/equivalent ones.

**Validation gate:** sources→sinks map in `09-source-review/`; every confirmed dynamic finding has a source location and root cause; high-value source-only candidates queued for dynamic confirmation.

## 11. Phase 10 — Exploitation, Chaining & Impact Demonstration

**Goal:** Demonstrate real impact with the **minimum** action — and chain smaller issues into high-severity outcomes (the difference between a scanner list and a pentest).

### 11.1 Principles
- Minimal proof: RCE → `id`/`hostname` or OOB callback; data exposure → schema + row counts; auth bypass → reach the privileged view. No persistence, no live shells on prod, no bulk PII.
- **Chaining:** combine issues into an attack path — e.g., traversal → config file → admin creds → admin panel → RCE (OSWA), or auth-bypass SQLi → admin reset → SSTI → RCE (OSWE). The chain *is* the finding.
- Reversibility: if a write is required to prove impact, write a benign marker to an agreed location and remove it.

### 11.2 Capture
Per chain: each step's request/response, the resulting access, and a narrative. Store in `10-exploit/`.

**Validation gate:** each exploited finding/chain is reproducible with captured proof; target left in original state (markers removed).

## 12. Phase 11 — Reporting

**Goal:** Deliver evidence-based findings readable by a PM and defensible to engineers/developers, with focused remediation.

### 12.1 Report structure
1. Executive summary (themes, business risk, key chains).
2. Scope, methodology (WSTG-aligned), RoE, environment, WAF posture.
3. Findings (ranked), using the template below.
4. Notable attack chains (narrative + impact).
5. Coverage statement (what was and wasn't tested; WSTG categories covered).
6. Evidence appendix (cross-referenced to `07-evidence/`).
7. Remediation roadmap (developer-focused, prioritized).

### 12.2 Finding template
```markdown
### [Finding ID] <Title>

**Summary:** <Plain-language: what's wrong and why it matters.>

**Affected component:** <app / endpoint / parameter / function; URL>
**Attack surface / entry point:** <request that reaches it>
**Weakness classification:** <e.g., reflected XSS, BOLA, SQLi, SSTI, insecure deserialization>
**WSTG ID:** WSTG-XXXX-###     **OWASP Top 10:** A##:2021
**CWE mapping:** CWE-### <name>   (mark "uncertain" if not confidently mappable)
**CVSS:** <vector> → <score/severity>   (omit/mark "not scored" if unsupportable)

**Preconditions:** <Auth level, role, configuration, position needed.>

**Evidence:**
<Observed facts only — raw request/response excerpts, screenshots.
(White-box) source file:line of the vulnerable sink + tainted source.
Reference 07-evidence/ artifacts by filename + hash.>

**Root cause:** <Underlying defect: missing output encoding, no ownership check,
string-concatenated query, unsafe deserialization, etc. Cite source where available.>

**Impact:** <Concrete, evidence-bounded consequence. Separate CONFIRMED from INFERRED.>

**Exploitability:** <Reliability, preconditions, skill/access. CONFIRMED vs INFERRED.>

**Reproduction steps:**
1. <Exact request(s)/payload(s), in order.>

**Expected vs. observed:**
- Expected (secure): <encoded/rejected/authorized behavior>
- Observed: <what the app did>

**Fix recommendation:** <Specific, developer-actionable: parameterized queries,
server-side authorization at the data layer, context-aware output encoding,
safe deserialization / allow-lists, etc.>

**Validation after fix:** <The same request that succeeds pre-fix and fails post-fix.>

**Residual risk:** <Related endpoints/inputs not yet remediated; compensating controls.>
```

### 12.3 Quality checks
Each finding: evidence present; WSTG/CWE/OWASP supportable or marked uncertain; reproduction validated; access-control findings shown with two identities; no scanner-only claims; exec summary readable standalone.

**Validation gate:** every finding links to hashed evidence (+ source location where white-box); coverage by WSTG category stated; chains documented.

## 15. Appendix B — WSTG Coverage Checklist (per app)
```
[ ] INFO  Recon: tech, TLS, WAF, JS/endpoint intel
[ ] MAP   Content + params + APIs discovered; role-access matrix built
[ ] CONF  TLS, headers, methods, cookies, CORS, exposed files, errors
[ ] IDNT  User enumeration, provisioning, role assignment
[ ] ATHN  Auth bypass, spray (lockout-aware), reset logic, MFA, JWT
[ ] SESS  Token entropy, fixation, lifetime, logout, CSRF
[ ] ATHZ  BOLA/IDOR, BFLA, privesc, traversal, mass assignment (≥2 identities)
[ ] INPV  SQLi, NoSQLi, cmdi, SSTI, XSS (S/R/DOM), SSRF, XXE, LFI/upload, CRLF, proto pollution, deser
[ ] BUSL  Logic bypass, race conditions, workflow auth
[ ] CLNT  DOM XSS, postMessage, clickjacking, client storage
[ ] API   REST/GraphQL/WS — authz re-tested at API layer
[ ] WBOX  (if source) sources→sinks mapped; root cause per finding
[ ] EXP   Chains demonstrated with minimal impact proof
```

## 16. Appendix C — Safe-PoC Guidance
- **RCE:** `id`/`hostname` or OOB DNS/HTTP callback. No live shells/persistence on prod.
- **SQLi:** `current_user`, version, single-row or table-name + count. No `--dump` of sensitive data on prod.
- **XSS:** benign marker (`alert(document.domain)`), screenshot. Never target real users' sessions.
- **SSRF/XXE:** OOB callback + read-only internal/metadata proof. No internal writes.
- **File ops:** benign marker file to an agreed path, removed after. No web shells on prod.
- **Deserialization:** OOB callback / benign command + documented gadget. No destructive payloads.
- **Account lockout:** model policy first; spray within budget; track per-account attempts.
