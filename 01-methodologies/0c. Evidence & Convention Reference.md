# Evidence & Convention Reference
### Single source of truth — Master Index §5.2. When a playbook drifts from this, this file wins.

This file defines the conventions every engagement in the suite assumes: how evidence is named, hashed, and separated from interpretation. It is dropped into each engagement's evidence directory by `engagement-init.sh`.

---

## 1. Evidence Naming

```
<phase>_<target-or-asset>_<tool>_<UTC>.<ext>
```
- **phase** — the numeric phase the artifact came from (e.g., `03`).
- **target-or-asset** — host/IP/endpoint/account/component (sanitize for filesystem).
- **tool** — the tool that produced it (e.g., `nmap`, `smbmap`, `nuclei`, `frida`).
- **UTC** — collection time, `YYYYMMDDTHHMMZ` (always UTC, never local).
- **ext** — true file type (`txt`, `json`, `xml`, `pcap`, `png`, `har`, `log`).

**Example:** `03_198.51.100.10_smbmap_20250625T1432Z.txt`

---

## 2. Evidence Integrity

- Hash **raw** evidence at collection time; never edit raw files.
- Store raw output under `07-evidence/raw/`; screenshots under `07-evidence/screenshots/`; captures (pcap/har/traffic) under their subdir.
- Each finding cites its artifacts by **filename + sha256**.

Generate/refresh the manifest:
```bash
# from the engagement root
find 07-evidence -type f ! -name SHA256SUMS -exec sha256sum {} \; | sort -k2 > 07-evidence/SHA256SUMS
# verify later
sha256sum -c 07-evidence/SHA256SUMS
```
(The `hash-evidence.sh` helper seeded at the engagement root does this for you.)

---

## 3. Evidence vs. Hypothesis (mandatory separation)

- **Evidence** = what was *observed* (the request/response, the sanitizer abort, the debugger state, the response diff). Goes in the finding's **Evidence** field and in `07-evidence/`.
- **Hypothesis** = what you *infer* it means (exploitability, impact). Goes in **Impact**/**Exploitability**, explicitly labeled.
- Never assert impact you did not observe.

## 4. CONFIRMED vs. INFERRED

For exploitability and impact, state which:
- **CONFIRMED** — demonstrated with a safe PoC / captured artifact.
- **INFERRED** — reasoned but not safely demonstrable in scope; document the blocking constraint and **what would confirm it**. Do not fabricate a result.

## 5. Mapping Uncertainty

Mark CWE / CVSS / MITRE ATT&CK (and domain overlays: WSTG / OWASP API / MASVS / tiering) **"uncertain"** rather than guessing. A defensible "uncertain" beats a fabricated mapping.

---

## 6. Safe-PoC Baseline (applies across the suite)

Prove the **primitive**, then stop. Benign markers, read-only proof, out-of-band (OOB) callbacks, reversible writes (marker removed). No persistence, stealth, credential theft, malware behavior, or unauthorized access beyond the demonstration. No live shells or destructive payloads on production. Scope all proof to owned/authorized assets — and, for multi-tenant work, to your own test tenants. (Full per-impact-class table: Vulnerability Lifecycle playbook §5.2 / Appendix C.)

## 7. State Changes & Cleanup

Engagements that create state (Internal, AD/Windows-Enterprise, and the data-handling side of Social Engineering) maintain a `cleanup/change-log/changes.md` (or `10-data-handling/`) from the **first** modification, and revert every entry with evidence before close. Each logged change records: what, where, when, why, and how it was reverted.

---

*Conventions are stated once here and in the Master Index so the ten playbooks stay consistent. Keep raw evidence immutable and hashed, keep observation separate from inference, prove impact with the minimum safe action, and log/revert everything you create.*
