# API & Microservices Penetration Testing Methodology

**Document type:** Repeatable assessment playbook (black-box + white-box / schema- and source-assisted)
**Scope of applicability:** REST/GraphQL/gRPC/SOAP/WebSocket APIs and the microservice architectures behind them — gateways, service mesh, inter-service auth, message/event buses, and (where in scope) the container/orchestration and CI/CD substrate
**Standards referenced:** OWASP API Security Top 10 (2023), OWASP WSTG v4.2, OWASP ASVS, MITRE ATT&CK (Enterprise + Containers), CWE, CVSS v3.1/v4.0, NIST SP 800-115/800-204 (microservices security)
**Toolchain assumed:** This Kali build — Burp/ZAP/Caido + `mitmproxy` (proxy), `ffuf`/`feroxbuster`/`arjun` (discovery), `nuclei` (templates incl. exposures/k8s), `sqlmap`/`commix`/`sstimap` (injection), `wscat`/`websocat` (WS), `hashcat -m 16500` (JWT), `peirates` (K8s), `trivy`/`syft`/`cosign` (image/SBOM/supply chain), `gitleaks`/`trufflehog` (secrets), `pacu`/`cloud-enum`/`s3scanner` (cloud), `ligolo-ng`/`proxychains4` (pivot). `jwt_tool`, `kiterunner`, `grpcurl`, `graphw00f`/`InQL`, `interactsh`/Burp Collaborator referenced where relevant (install / Burp Pro).

> **Operating constraints baked into this methodology**
> - **SSRF is the lateral-movement primitive.** In microservices, a single SSRF often pivots east-west into "internal-only" services that trust any caller. Treat every URL-consuming field as a potential gateway into the mesh.
> - **The gateway is not the perimeter.** Test services both *through* the API gateway and (where reachable) *directly*, since access control, rate limiting, and identity are frequently enforced only at the edge.
> - **Identity is propagated, and propagation is trust.** Internal services commonly trust headers (`X-User-Id`, `X-Forwarded-For`, `X-Internal`) set upstream. Test whether those can be spoofed by a direct or relayed caller.
> - **Async has side effects.** Message-queue/event/webhook testing can trigger real downstream actions (payments, emails, provisioning). Use test data and benign markers; coordinate before injecting into live buses.
> - **Resource-consumption and orchestration tests are availability- and authorization-sensitive.** Rate-limit/DoS testing, Kubernetes API access, and cloud metadata require explicit RoE and (for managed clusters) cloud-provider authorization.
> - **Multi-identity is mandatory for authorization.** BOLA/BOPLA/BFLA need ≥2 accounts per role and cross-tenant pairs. Map findings to **OWASP API Top 10 + CWE + ATT&CK**; separate evidence from hypothesis.

## 0. Phase 0 — Pre-Engagement & Scoping

**Goal:** Fix authorization, the service/API inventory boundary, identities, and the architectural/RoE safety rails.

### 0.1 Authorization & scope (hard blocker) — `00-scope/`
1. Signed authorization naming in-scope API base URLs, service names, gateways, and (if applicable) cluster/namespace and cloud account IDs with ownership attestation.
2. **Environment:** prod vs staging; prefer non-prod for resource-consumption, async side-effect, and orchestration testing. Record versions/commits if source-backed.
3. **Exclusions:** third-party APIs you only *consume* (need their authorization), payment/identity providers, shared platform services, fragile event consumers.
4. **Schemas/specs:** OpenAPI/Swagger, GraphQL SDL, `.proto` files, AsyncAPI, Postman/Bruno collections — request all available; they are the master test surface.
5. **Identities:** accounts for **every role** and **≥2 tenants** (cross-tenant isolation testing), plus any service-to-service credentials/API keys in scope. Test data that may be created/modified.
6. **Architecture inputs:** gateway(s), service mesh (Istio/Linkerd), auth model (OAuth2/OIDC/mTLS/API keys/HMAC), message bus (Kafka/RabbitMQ/SQS), and whether the container/orchestration/CI-CD layer is in scope.
7. **RoE intensity:** automated scanning against prod? rate-limit/DoS testing? injecting into live queues? K8s API interaction? cloud metadata? Real vs marker data for exfil proof.
8. **Cloud/cluster authorization:** for managed K8s/serverless, confirm the provider's testing policy is honored.
9. Emergency stop + contacts (incl. platform/SRE deconfliction).

### 0.2 Decision point
- No orchestration/cloud RoE → cap at API + microservice-application layer (Phases 1–6, 8–9); document infra surface as out-of-scope observations.
- Prod-only → non-destructive PoC for injection/SSRF; flag where deeper proof needs staging.

**Validation gate:** authorization stored; schemas collected; multi-tenant/multi-role identities received; architecture + RoE for async/resource/orchestration agreed.

## 2. Phase 1 — Architecture Mapping & Discovery

**Goal:** Enumerate every service, endpoint, protocol, and trust boundary. The architecture map drives every later phase. MITRE ATT&CK: Reconnaissance (TA0043).

### 2.1 Schema-driven inventory (richest source)
```bash
# Pull and parse machine-readable specs into an endpoint inventory
curl -s https://api.target/openapi.json -o 01-arch/schemas/openapi.json
curl -s https://api.target/swagger.json -o 01-arch/schemas/swagger.json
# GraphQL introspection (if enabled) -> full schema
# graphw00f to fingerprint engine; InQL/clairvoyance to recover schema
# gRPC: server reflection -> grpcurl -plaintext host:port list   (then describe each service)
```
Build `01-arch/inventory/endpoints.csv`: per endpoint record method(s), auth requirement, parameters, object references, and owning service.

### 2.2 Content & verb discovery (when specs are incomplete)
```bash
# API-aware content discovery
ffuf -u https://api.target/FUZZ -w /usr/share/seclists/Discovery/Web-Content/api/api-endpoints.txt -mc all -o 01-arch/inventory/ffuf_api.json
# kiterunner (install) for route+method bruteforce from API wordlists/Swagger patterns
# Verb tampering: an endpoint that 403s on GET may allow PUT/DELETE/PATCH — enumerate hidden methods
for m in GET POST PUT PATCH DELETE OPTIONS HEAD; do curl -s -o /dev/null -w "$m %{http_code}\n" -X $m https://api.target/resource; done
```

### 2.3 Protocol & topology matrix
Record per service: protocol (REST/GraphQL/gRPC/SOAP/WS), edge vs internal, gateway in front, auth model, and dependencies (which services it calls). Draw the **trust-boundary diagram**: client → gateway → services → data stores / buses / external. Mark each boundary's intended authentication.

### 2.4 Gateway vs direct reachability
Determine whether internal services are reachable without traversing the gateway (different port, internal DNS, exposed pod/NodePort, or via SSRF later). This seeds Phase 5 gateway-bypass testing.

**Validation gate:** endpoint inventory + protocol matrix + trust-boundary diagram complete; gateway/direct reachability noted per service.

## 3. Phase 2 — Authentication & Token Architecture

**Goal:** Test how identity is established and propagated across the system. OWASP API2 (Broken Authentication). MITRE ATT&CK: T1078, T1550.

### 3.1 Edge authentication
- OAuth2/OIDC flow review: authorization-code/PKCE vs implicit; `redirect_uri` validation; `state`/nonce; token leakage; scope enforcement; consent/app-grant abuse.
- API keys: entropy, transport, scoping, revocation, rate binding.
- Session/cookie auth (if hybrid): see web methodology (SESS).

### 3.2 JWT deep test
```bash
jwt_tool <token> -M at -t https://api.target/...     # alg confusion, claim tampering (install)
hashcat -m 16500 jwt.txt wordlist.txt                # weak HMAC secret
```
- `alg:none`, HS/RS confusion (sign with public key as HMAC secret), `kid` injection (path/SQL), JWK/JKU header SSRF, expired/`nbf` not enforced, signature not verified, claim-based authorization trusted blindly (`role`, `tenant`, `iss`, `aud`).

### 3.3 Service-to-service auth
- mTLS: is client-cert actually required and validated on internal endpoints, or only at the edge?
- Internal/service tokens, HMAC request signing: replayability, scope, key management.
- **Identity propagation trust (critical):** does an internal service trust headers like `X-User-Id`, `X-Roles`, `X-Tenant-Id`, `X-Forwarded-*`, `X-Internal-Request` set upstream? Test by sending them on a direct/relayed request (Phase 5.2).

**Validation gate:** edge and service-to-service auth tested; JWT/key/mTLS weaknesses captured; identity-propagation trust assumptions identified for Phase 5.

## 4. Phase 3 — Authorization (Object / Property / Function Level)

**Goal:** Prove access-control failures with **multiple identities and tenants**. OWASP API1/API3/API5. *Highest-value, most-missed category.*

### 4.1 BOLA / IDOR — object level (API1)
For each object reference (ID, UUID, slug, filename, composite key), replay user A's request substituting user B's / another tenant's identifier; confirm cross-user/cross-tenant access. Automate with Burp Autorize/Auth Analyzer (request as A, replay as B, diff). gRPC/GraphQL object args are equally in scope.

### 4.2 BOPLA — object property level (API3)
- **Mass assignment / autobinding:** add privileged properties (`role`, `isAdmin`, `tenantId`, `verified`, `balance`) to create/update bodies.
- **Excessive data exposure:** does the response return more fields than the UI shows (server returns full object, client filters)? Capture sensitive fields leaked in the raw response.

### 4.3 BFLA — function level (API5)
- Access admin/privileged operations from a lower-privilege identity (vertical).
- Force-call administrative methods from the inventory (incl. verb-tampered methods from 2.2); verify **server-side** enforcement, not gateway/UI hiding.
- Cross-role method matrix: attempt every privileged endpoint from every lower role.

**Validation gate:** each access-control finding demonstrated with ≥2 identities (and cross-tenant where multi-tenant); response diff captured proving unauthorized access/property/function.

## 5. Phase 4 — API Input Validation & Injection

**Goal:** Find injection and data-handling flaws across protocols. **Non-destructive PoC; OOB over data damage.**

### 5.1 Per-protocol injection surface
- REST/JSON: SQLi/NoSQLi (operator injection `$ne`/`$gt`), command injection, SSTI (if server-rendered), header/host injection.
- GraphQL: injection through resolver arguments; introspection-aided field probing; alias/batch abuse (also Phase 6).
- gRPC/protobuf: malformed/oversized fields, enum/oneof edge cases, type confusion; fuzz the message via `grpcurl` with crafted JSON→proto.
- SOAP/XML: XXE (external entity → file read / SSRF), parameter tampering.
- WebSocket: message-level injection and authorization (`wscat`/`websocat`).
```bash
sqlmap -r 07-evidence/raw/req.txt --batch --level 3 --risk 2 --output-dir=04-input/sqli   # prove, don't --dump on prod
commix -r 07-evidence/raw/req.txt --output-dir=04-input/cmdi                              # OOB-confirm, minimal proof
```

### 5.2 SSRF discovery (the microservices keystone — leads to Phase 5)
Identify every URL/host-consuming field (webhooks, importers, image/PDF/preview generators, callback URLs, federation metadata, "fetch from URL"). Confirm via OOB callback; this is the entry to east-west pivoting.

**Validation gate:** injection confirmed (not scanner-only) with non-destructive PoC; all SSRF-capable inputs catalogued for Phase 5.

## 6. Phase 5 — Microservices Architecture Attacks

**Goal:** Exploit the architecture, not just a single endpoint. This is what distinguishes microservices testing from single-app API testing.

### 6.1 SSRF → east-west pivot (OSWE-style chain)
From a confirmed SSRF (5.2), escalate **read-only** through the internal network the service can reach:
1. Loopback/internal reach: hit `127.0.0.1`, internal DNS names, service-mesh sidecar ports.
2. **Port/subnet scanning via SSRF:** enumerate internal services and ports through response/timing differences.
3. **Host enumeration:** identify internal services (admin panels, management APIs, dashboards).
4. **Cloud metadata** (if cloud-hosted and in scope): read instance metadata / service-account tokens — capture proof, do not abuse beyond demonstration.
5. Alternative URI schemes (`file://`, `gopher://`, `dict://`) where the client library allows.

### 6.2 Inter-service trust & identity spoofing
- Send forged identity-propagation headers (`X-User-Id`, `X-Roles`, `X-Tenant-Id`, `X-Internal-Request`) on direct or SSRF-relayed requests to internal services; confirm whether the service authorizes based on them without re-validating.
- Test whether internal services require any authentication at all ("internal = trusted" assumption).

### 6.3 Gateway bypass
- Reach a service **directly** (internal port, NodePort, internal DNS, pod IP via SSRF) and replay requests that the gateway would have authenticated/rate-limited/filtered. Confirm controls enforced only at the edge are absent on the service.

### 6.4 Internal / admin / management API exposure
- Hunt management surfaces: Spring Boot `/actuator` (env/heapdump/mappings), Kong/other gateway admin APIs, service dashboards, debug/metrics endpoints, message-broker management consoles.
- OSWE pattern: reachable **admin API auth bypass → configuration change → RCE** (e.g., Kong admin API). Demonstrate the primitive minimally.

### 6.5 Async / event-driven surfaces
- Message queues / event streams (Kafka/RabbitMQ/SQS): can you publish to topics consumers trust? Are messages authenticated/validated by consumers? **Use test data + benign markers; do not trigger real downstream actions.**
- Webhooks: SSRF via webhook target, signature validation bypass, replay.

### 6.6 Secrets exposure
- Hard-coded secrets/tokens in responses, error messages, debug endpoints, JS/spec files, or container/IaC (Phase 7). Service-account tokens reachable via SSRF/metadata.

**Validation gate:** architecture-level findings (SSRF pivot path, gateway bypass, trusted-header spoof, exposed admin API) demonstrated with captured proof; the trust-boundary diagram annotated with each broken boundary.

## 7. Phase 6 — Resource Consumption & Business-Flow Abuse

**Goal:** Test availability and high-value-flow controls. OWASP API4/API6. **Availability-sensitive — RoE-gated, careful, non-prod where possible.**

### 7.1 Unrestricted resource consumption (API4)
- Missing/weak rate limiting, pagination limits, file-size/field-count limits, expensive operations.
- GraphQL: deeply nested queries, alias/batch amplification, complexity/depth DoS — test conservatively to *demonstrate* the absence of limits, not to cause outage.

### 7.2 Sensitive business-flow abuse (API6)
- Automatable high-value flows (purchases, signups, OTP requests, comment/spam): can they be scripted past the intended rate/human-interaction controls?

### 7.3 Race conditions
- Concurrent requests to double-spend, bypass one-time-use tokens, or exploit TOCTOU (Burp Turbo Intruder / parallel clients).

**Validation gate:** resource/business-flow findings shown with the minimal evidence proving the missing control; no sustained outage caused.

## 8. Phase 7 — Container, Orchestration & Supply-Chain Surface *(if in scope)*

**Goal:** Assess the substrate microservices run on. MITRE ATT&CK: Containers (TA0007/Privilege Escalation). **Requires explicit RoE + cloud-provider authorization for managed clusters.**

### 8.1 Container images & SBOM
```bash
trivy image <registry>/<image>:<tag>          # vulns, misconfigs, secrets in layers
syft <image> -o table                          # SBOM / component inventory
cosign verify <image>                          # image signing/provenance (if used)
```

### 8.2 Kubernetes / orchestration
```bash
# From an in-scope foothold pod or with provided kubeconfig:
peirates                                        # K8s pentest: service-account abuse, RBAC, secrets, lateral
# Manual checks: reachable kube-apiserver, kubelet (10250), etcd; service-account token in pod
#   (/var/run/secrets/kubernetes.io/serviceaccount/token); RBAC of that token; pod security context.
nuclei -t http/exposures -u https://<cluster-endpoint>   # exposed dashboards/APIs
```
- Service-account RBAC over-permission, mountable secrets, privileged/hostPath pods, exposed kubelet/etcd, namespace isolation, network policy gaps. Demonstrate access; don't backdoor workloads unless RoE explicitly authorizes.

### 8.3 CI/CD & IaC
```bash
gitleaks detect -s <repo>                       # secrets in history
trufflehog filesystem <repo>                    # verified secrets
trivy config <iac-dir>                          # IaC misconfig (Terraform/K8s manifests/Dockerfile)
```
- Pipeline injection, unsigned artifacts, over-privileged runners, secret leakage, dependency/supply-chain (SCA) risk.

### 8.4 Cloud edges (if cloud-hosted)
- Metadata/service-account exposure (often reached via SSRF in Phase 5), storage misconfig (`s3scanner`), IAM over-permission (`pacu` for AWS) — within authorized accounts only.

**Validation gate:** infra findings captured with bounded proof; no workload backdooring/persistence unless RoE-authorized; cloud actions confined to authorized accounts.

## 9. Phase 8 — White-Box / Source- & Schema-Assisted Review

**Goal:** Root cause for every dynamic finding and discovery of what black-box misses across service boundaries. Runs in parallel with Phases 1–7.

### 9.1 Cross-service sources → sinks
Trace attacker-controlled input from edge through gateway and across service calls to sinks (queries, command exec, deserializers, HTTP clients = SSRF, template render, file ops). A taint path that crosses a service boundary where the downstream service assumes "trusted internal input" is a high-value finding.

### 9.2 Identity-propagation & authorization review
- Where is authn/authz enforced — gateway only, or per service? Find services that trust upstream headers/claims without re-validation (root cause of 6.2).
- Object-ownership and tenant checks at the data layer (root cause of BOLA/BOPLA).

### 9.3 Schema-driven test generation
Generate negative/abuse cases directly from OpenAPI/GraphQL SDL/`.proto`: every parameter at its boundary, every object reference cross-identity, every method via verb tampering. Drive coverage from the schema so no endpoint is skipped.

### 9.4 Secrets & dependency review
```bash
graudit -d default ./src
gitleaks detect -s ./src
# semgrep --config auto ./src   (install) for taint rules across services
```

**Validation gate:** sources→sinks map (cross-service) in `08-source-review/`; every confirmed dynamic finding has a source location + root cause; schema-derived gaps queued for dynamic confirmation.

## 10. Phase 9 — Exploitation & Chaining

**Goal:** Demonstrate real impact with minimal action; chain across services (the deliverable).

### 10.1 Principles
- Minimal proof: RCE → `id`/OOB callback; data exposure → schema + counts; cross-tenant → reach another tenant's object; SSRF → read-only internal/metadata proof. No persistence, no live shells on prod, no bulk PII.
- **Chaining across services** (the microservices signature): e.g., *verb-tampering discovers an endpoint → mass assignment sets a privileged property → BOLA reaches another tenant's object*; or *SSRF → internal port scan → exposed admin API → auth bypass → config change → RCE* (OSWE Kong pattern); or *forged `X-User-Id` to a gateway-bypassed internal service → privileged action*. The chain is the finding.

### 10.2 Capture
Per chain: each request/response, the resulting access, and a narrative annotated on the trust-boundary diagram. Store in `09-exploit/`.

**Validation gate:** each exploited finding/chain reproducible with captured proof; target left in original state (markers removed; no live queue side effects).

## 11. Phase 10 — Reporting

**Goal:** Evidence-based findings readable by a PM and defensible to engineers, with architecture context and developer-focused remediation.

### 11.1 Report structure
1. Executive summary (themes, business risk, key cross-service chains).
2. Scope, methodology, RoE, environment, architecture overview + trust-boundary diagram.
3. Findings (ranked), using the template below.
4. Notable attack chains (narrative + diagram + impact).
5. Coverage statement (services/protocols/OWASP API categories covered; infra in/out of scope).
6. Evidence appendix (cross-referenced to `07-evidence/`).
7. Remediation roadmap (edge vs per-service authz, schema validation, rate limiting, secret management, RBAC hardening).

### 11.2 Finding template
```markdown
### [Finding ID] <Title>

**Summary:** <Plain-language: what's wrong and why it matters.>

**Affected component:** <service / endpoint / method / property; URL or internal service>
**Attack surface / entry point:** <edge endpoint, internal service, queue, or chain step>
**Weakness classification:** <e.g., BOLA, BFLA, mass assignment, SSRF, broken service-auth, exposed admin API>
**OWASP API Top 10:** API##:2023     **WSTG/relevant ID:** <if applicable>
**CWE mapping:** CWE-### <name>   (mark "uncertain" if not confidently mappable)
**MITRE ATT&CK:** <Txxxx(.xxx)>
**CVSS:** <vector> → <score/severity>   (omit/mark "not scored" if unsupportable)

**Preconditions:** <Auth level, role/tenant, network position (edge vs internal), config.>

**Evidence:**

**Reproduction steps:**
1. <Exact request(s)/chain, in order, including identities/tenants used.>

**Expected vs. observed:**
- Expected (secure): <per-service authz / validated identity / rejected>
- Observed: <what the system did>

**Fix recommendation:** <Specific: enforce authz at each service, validate propagated identity

### 11.3 Quality checks
Each finding: evidence present; OWASP API/CWE/ATT&CK supportable or marked uncertain; reproduction validated; access-control findings shown with multiple identities/tenants; no scanner-only claims; chains annotated on the diagram.

**Validation gate:** every finding links to hashed evidence (+ source location where white-box); coverage by service/protocol/OWASP-API category stated; chains documented.

```

## 15. Appendix C — Microservices Trust-Boundary Checklist
```
[ ] Each service authenticates callers (not "internal = trusted")
[ ] Authz enforced per service, not only at the gateway
[ ] Propagated identity (X-User-Id/X-Roles/X-Tenant/JWT claims) re-validated downstream
[ ] Services NOT directly reachable bypassing the gateway (or controls replicated if they are)
[ ] No exposed admin/management APIs (actuator, gateway admin, broker consoles)
[ ] mTLS/service tokens actually enforced on internal endpoints
[ ] Outbound URL handling allow-listed (SSRF-resistant)
[ ] Message-bus producers/consumers authenticate + validate messages
[ ] Secrets not in responses/errors/specs/images/IaC; service-account tokens scoped
[ ] Tenant isolation holds across every object reference
```

## 16. Appendix D — Safe-PoC Guidance
- **SSRF:** OOB callback + read-only internal/metadata proof (capture the response). No internal writes, no abusing harvested service-account tokens beyond demonstration.
- **RCE (admin API / injection):** `id`/`hostname` or OOB callback; no persistence/live shells on prod.
- **BOLA/cross-tenant:** retrieve another tenant's object identifier + a non-sensitive field as proof; do not bulk-export.
- **Injection:** `current_user`/version/single row; no `--dump` of sensitive data on prod.
- **Async:** test topics/queues with marker payloads to controlled consumers; never trigger real downstream actions (payments, emails, provisioning).
- **Resource/DoS:** demonstrate the missing limit with a bounded burst; never sustain an outage.
- **K8s/cloud:** enumerate and prove access; no workload backdoor/persistence unless RoE-authorized.

---

*This methodology is bounded to authorized testing. Treat SSRF as the lateral primitive, test services through and around the gateway, never trust propagated identity at face value, and chain issues across services to show real risk. Use test data for async surfaces, keep resource and orchestration testing within RoE and provider policy, and map each finding to OWASP API Top 10 + CWE + ATT&CK with evidence and root cause. Mark mapping and exploitability uncertainty explicitly.*
